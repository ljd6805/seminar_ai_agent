- 브라우저 렌더 확인: MCP playwright는 `file:` 차단 → `python3 -m http.server` 로 서빙 후 접속. **편집 후엔 캐시 회피를 위해 `?v=N` 쿼리로 재로드**(해시만 바꾸면 재로드 안 됨). **애니메이션 대기는 Bash sleep 말고 playwright의 wait(browser_wait_for) 사용** — 백그라운드 탭 타이머 스로틀로 setTimeout이 밀림. 스크린샷은 cwd(repo 루트)에 저장되니 리뷰 후 삭제.
- 문서는 개념 수준이며 버전 표기(vX.Y)로 관리. 갱신 시 버전과 아래 상태를 함께 업데이트.

## 확정된 결정

- **(v0.9) 척추 = A "알고 써보자"** — "AI agent가 뭐고 우리 환경에서 어떻게 쓰는지 알고, 사내에서 하나 써보자". B "일하는 방식을 바꿔야 한다"는 주장하지 않음(청중이 결론내리게) — **17장 클로징 미래형 한 줄로만** 존재. 표지 제목은 "오늘부터 쓸 수 있습니다"(B 선언 "일하는 방식이 바뀝니다" 폐기). (2026-07-07 결정)
- **(v0.9) 구조**: 데모1을 정의 앞으로(show→tell) / 구 6+7·구 11+12 병합 / 부록 슬라이드 신설 / 섹션 브리지(5→6, 10→11) / 보안 원칙은 6장에서 1회 온전 진술 + 이후 콜백만.
- **(v0.9) 표기 규칙**: MCP는 대문자(약어), skill·plugin은 소문자. 정준 나열 순서 **skill · MCP · plugin**(능력→연결→배포). 청중 노출 데모 라벨은 **데모 1/2/3**(내부 관리번호 Ex1/Ex2/Ex4 유지, Ex3=GitOps 결번은 비노출). 영어 장식 라벨(SAME INPUT·auto) 한글화. '클라이맥스'→'정점'.
- 핵심 서사: AI agent가 있다 → 장점 → 이렇게 쓰면 된다 (첫 소개). 청자는 이해도 낮음 → 도식 중심, 전문 용어는 쉬운 정의로 받쳐줌(용어는 괄호 보조, 강조는 '무엇이 가능해졌나').
- **우산 용어: "확장 요소"**(= skill·MCP·plugin). "부품"은 plugin이 그릇이라 어긋나 폐기 (2026-07-06 결정)
- 데모는 라이브가 아니라 **미리 만든 예시**를 자료에 삽입. 데모2·데모3은 **하나의 "문서 parsing" 작업을 공유**(데모2=기본 동작, 데모3=harness 유무 대조).
- 데모3 대조 축은 **품질 + 재현성** — 재현성은 "스키마 일치/같은 구조"까지만 (속도·퍼센트 등 성능 수치는 쓰지 않음).
- harness는 **verification harness(UVM)** 에 빗대는 재구성 + **공식 근거**(Claude Code 문서의 "agentic harness") 명시. 스키마=assertion(형식 게이트), 내용 정합은 별도 검사라는 한계도 명시(정직성 — 그 자리를 지금은 사람이 = CTA의 사람 검수 루프와 연결).
- GitOps 협업 소재는 제외. 설치·주의는 11장에 한 줄씩 흡수, 전체 도구 목록은 부록(19장)으로 실체화.
- **(2026-07-07) skill·MCP·plugin 각 1장 할당**: 의미·목적·구조·동작원리(plugin은 동작 대신 관계·설치)를 장마다 — 구조는 파일트리 터미널, 동작은 ①②③ 스텝 박스.
- 마무리 3장 골격(정점 → CTA → Q&A) 유지 — 정점은 데모1 정지컷 1회만(대화 문구는 데모1과 글자 그대로 동일), CTA는 사내 LLM·스스로 판단 가능한 작은 작업·사람 검수 루프.

## 작업 시 주의 (가드레일)

- **데모 예시는 "구성된 illustrative 사례"** (`docs/seminar-page-list.md` 부록 A). 참인 메커니즘(스키마 검증 + 재시도가 품질·재현성을 높임)을 대표 사례로 보여주는 것으로, 입문 세미나에서 **그대로 제시해도 정당함**. 실패 3종(필드 누락/키 드리프트/preamble)은 벤더 공식 문서가 인정하는 실제 현상. 사용자가 원하면 자신의 harness 실제 실행 결과로 교체 가능(선택 사항).
- 다만 이 구성된 예시를 **"특정 harness를 실제 측정한 수치"인 것처럼 라벨링하지 말 것.** 속도·퍼센트 등 성능 숫자도 쓰지 않기로 함(품질·재현성으로만).
- 재현성 주장은 "스키마 일치/같은 구조"까지만 — "매번 동일한 JSON" 같은 값 결정론 표현 금지 (v0.9에서 교정됨, 재발 주의).
- 시각 표현·디자인·애니메이션 구체화는 아직 보류 상태. 사용자가 넘어가자고 하기 전엔 개념 수준 유지.
- 사용자는 한 번에 다 만들기보다 대화로 얼개를 잡아가는 방식을 선호. 큰 결정은 먼저 제안하고 확인받기. (2026-07-07 세션에서 장별 문구·구조는 일괄 위임받아 반영 — 이후 새 방향 전환은 다시 확인받을 것.)
