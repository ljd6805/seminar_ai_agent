- 브라우저 렌더 확인: MCP playwright는 `file:` 차단 → `python3 -m http.server 8765` 로 서빙 후 접속. **편집 후엔 캐시 회피를 위해 `?v=N` 쿼리로 재로드**(해시만 바꾸면 재로드 안 됨). **애니메이션 대기는 Bash sleep 말고 playwright의 wait(browser_wait_for) 사용** — 백그라운드 탭 타이머 스로틀로 setTimeout이 밀림. 스크린샷은 cwd(repo 루트)에 저장되니 리뷰 후 삭제.
- 문서는 개념 수준이며 버전 표기(vX.Y)로 관리. 갱신 시 버전과 아래 상태를 함께 업데이트.

## 확정된 결정

- **(v1.0, 2026-07-07) 90분 · 42장 확장** — A 서사 구멍(문제→해결 순서: 19장 맨몸 한계 / 신뢰 캘리브레이션: 30장 실패의 현실 / 용어 선행: 5장 LLM 기초 / 데모 중간 고리: 26장 SKILL.md / CTA 구체화: 39장 첫 프롬프트), B 압축 해체(역사 2장, skill·MCP 각 2장, 관계 정리 독립), C 도메인 공명(11장 설계·검증 pain, 32장 UVM 병렬 다이어그램, 16–17장 사내 도구·판단 사례). 간지 3장(14/18/29)·목차(2)·챗봇 vs agent(9)·위임 기준(13)·언제 쓰나(35)·FAQ(38)·용어집(42) 신설.
- **(v0.9) 척추 = A "알고 써보자"** — B "일하는 방식을 바꿔야 한다"는 주장하지 않음(청중이 결론내리게) — **40장 클로징 미래형 한 줄로만** 존재. 표지 제목 "오늘부터 쓸 수 있습니다".
- **(v0.9) 표기 규칙**: MCP는 대문자(약어), skill·plugin은 소문자. 정준 나열 순서 **skill · MCP · plugin**(능력→연결→배포). 청중 노출 데모 라벨은 **데모 1/2/3**(내부 관리번호 Ex1/Ex2/Ex4 유지, Ex3=GitOps 결번은 비노출). 영어 장식 라벨 한글화. '클라이맥스'→'정점'.
- 핵심 서사: AI agent가 있다 → 장점 → 이렇게 쓰면 된다 (첫 소개). 청자는 이해도 낮음 → 도식 중심, 전문 용어는 쉬운 정의로 받쳐줌(용어는 괄호 보조, 강조는 '무엇이 가능해졌나').
- **우산 용어: "확장 요소"**(= skill·MCP·plugin). "부품"은 plugin이 그릇이라 어긋나 폐기 (2026-07-06 결정)
- 데모는 라이브가 아니라 **미리 만든 예시**를 자료에 삽입. 데모2·데모3은 **하나의 "문서 parsing" 작업을 공유**(데모2=기본 동작 — v1.0부터 26장 SKILL.md 실물 + 27장 실행 2장, 데모3=harness 유무 대조).
- 데모3 대조 축은 **품질 + 재현성** — 재현성은 "스키마 일치/같은 구조"까지만 (속도·퍼센트 등 성능 수치는 쓰지 않음). v1.0부터 bad run을 실제 깨진 출력 스냅샷으로 노출.
- harness는 **verification harness(UVM)** 에 빗대는 재구성 + **공식 근거**(Claude Code 문서의 "agentic harness") 명시. v1.0에서 32장 UVM 병렬 다이어그램으로 확장(sequence→DUT→assertion ↔ 입력·절차→모델→스키마 검증). 스키마=assertion(형식 게이트), 내용 정합은 별도 검사라는 한계도 명시(그 자리를 지금은 사람이 = CTA의 사람 검수 루프와 연결).
- **(2026-07-07) 15장(구 7장) 우리 환경 = 격리 다이어그램**: 사용자 노드에서 두 갈래 분기 — 위(초록) 사내 AI agent(OpenCode·Roo·DS assistant API, 사내 직접 서빙) / 아래(경고색) 빗금 "회사 경계" 벽을 관통해 사외 AI agent(Claude, 도입 예정). 사용자 = 무엇을 어디로 보낼지 결정. **보안 원칙 1회 온전 진술 + 이후 콜백만.**
- **(2026-07-07) 12장(구 6장) "일의 재배치"**: 기존(전 과정 사람) vs 요구되는 방식(정의→agent 실행→확인·판단) flow 대조. **톤 주의: B 선언("바꿔야 한다")이 아니라 기술(記述) — 척추 A 유지.**
- 17장(보내도 되는 것) 판단 사례는 감 잡기용 예 — **"정확한 기준은 팀 보안 규정이 우선" 문구 필수** (슬라이드가 보안 정책을 창설하지 않음).
- **(2026-07-07, v1.1) "발표자료 제작" 예시 폐기** — 발표자의 일이지 청중의 일이 아니라는 사용자 판단. 데모 1(Ex1) 소재 = **회귀 로그 실패 분류 장면**(구성된 예시 라벨을 슬라이드에 명시, 채팅 오토플레이·성장 패널 재사용). 37장 데모 1 콜백("다시 그 장면") 폐기 → **"오늘 지나온 길" 회고 요약**(어디서·무엇으로·어떻게). 데모 번호 체계 불변.
- GitOps 협업 소재는 제외. 마무리 골격(회고 → FAQ → CTA → Q&A).

## 작업 시 주의 (가드레일)

- **데모 예시는 "구성된 illustrative 사례"** (`docs/seminar-page-list.md` 부록 A). 참인 메커니즘(스키마 검증 + 재시도가 품질·재현성을 높임)을 대표 사례로 보여주는 것으로, 입문 세미나에서 **그대로 제시해도 정당함**. 실패 3종(필드 누락/키 드리프트/preamble)은 벤더 공식 문서가 인정하는 실제 현상. 사용자가 원하면 자신의 harness 실제 실행 결과로 교체 가능(선택 사항).
- 다만 이 구성된 예시를 **"특정 harness를 실제 측정한 수치"인 것처럼 라벨링하지 말 것.** 속도·퍼센트 등 성능 숫자도 쓰지 않기로 함(품질·재현성으로만).
- 재현성 주장은 "스키마 일치/같은 구조"까지만 — "매번 동일한 JSON" 같은 값 결정론 표현 금지 (v0.9에서 교정됨, 재발 주의).
- 사내 도구(16장)·판단 사례(17장)의 세부 설명은 사용자 확인 전까지 일반적 서술 유지 — 사내 서비스의 실제 사양을 지어내지 말 것.
- 사용자는 한 번에 다 만들기보다 대화로 얼개를 잡아가는 방식을 선호. 큰 결정은 먼저 제안하고 확인받기. (2026-07-07: 42장 확장 + A/B/C 반영은 일괄 위임받아 반영 — 이후 새 방향 전환은 다시 확인받을 것.)
