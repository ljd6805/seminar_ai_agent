## 확정된 결정

- **(v1.3, 2026-07-07) plugin 발표 제외** — 본문 2 = **skill·MCP 둘만**, 각 요소를 "개념 → 구조·동작 → 데모"로 완결. plugin·관계 정리 장 삭제, 맨몸 한계 두 구멍(지식·연결), 용어집에서 plugin·hook 제거. **우산 용어 "확장 요소" = skill·MCP.** MCP 데모(데모 3) 신설 — 이슈 트래커 연결 장면(구성된 예시, 이슈 ID 가상). harness 데모는 **데모 4**로 재라벨. 내부 관리번호: Ex1(데모1)·Ex2(데모2)·Ex5(데모3, 신규)·Ex4(데모4), Ex3=GitOps 결번.
- **(v1.2) 역사 1장 병합 + "여기까지 온 길" 칩 폐기(→"도입 · 역사")** — 제목 "AI는 어떻게 agent가 되었나".
- **(v1.1) "발표자료 제작" 예시 폐기** — 데모 1 = **regression 로그 실패 분류 장면**(팀 용어 영문 표기, "구성된 예시 장면" 라벨 명시, **정적** — 애니메이션 제거). 콜백 장 폐기 → "오늘 지나온 길" 회고. **"월요일" 수사 표현 전면 제거**(데모 2 메모 속 마감일 데이터만 유지).
- **(v1.0) 90분 · 대확장** — A 서사 구멍(맨몸 한계/실패의 현실/LLM 기초/SKILL.md 실물/첫 프롬프트), B 압축 해체, C 도메인 공명(설계·검증 pain, UVM 병렬 다이어그램, 사내 도구·판단 사례).
- **(v0.9) 척추 = A "알고 써보자"** — B "일하는 방식을 바꿔야 한다"는 주장하지 않음 — **38장 클로징 미래형 한 줄로만**. 표지 제목 "오늘부터 쓸 수 있습니다".
- **(v0.9) 표기 규칙**: MCP는 대문자(약어), skill은 소문자. 정준 나열 순서 **skill · MCP**(능력→연결). 청중 노출 데모 라벨 **데모 1~4**. 영어 장식 라벨 한글화. '클라이맥스'→'정점'.
- 데모는 라이브가 아니라 **미리 만든 예시**. 데모 2(skill)·데모 4(harness)는 하나의 "문서 parsing" 작업 공유. 데모 4 대조 축 = **품질+재현성**, 재현성은 "스키마 일치/같은 구조"까지만(성능 수치 금지), bad run은 실제 깨진 출력 스냅샷.
- harness = **verification harness(UVM)** 비유 + 공식 근거("agentic harness"). 30장 병렬 다이어그램(sequence→DUT→assertion ↔ 입력·절차→모델→스키마 검증). 스키마=assertion(형식 게이트), 내용 정합은 사람이(CTA 검수 루프와 연결).
- **14장 격리 다이어그램**: 사용자 노드 두 갈래 분기(사내 초록/사외 경고색, 빗금 경계 벽). **보안 원칙 1회 온전 진술 + 콜백만.** 16장 판단 사례엔 **"정확한 기준은 팀 보안 규정이 우선" 문구 필수.**
- **11장 "일의 재배치"**: 기술(記述) 톤 — 척추 A 유지. GitOps 소재 제외. 마무리 골격(회고 → FAQ → CTA → Q&A).

## 작업 시 주의 (가드레일)

- **데모 예시는 "구성된 illustrative 사례"** (`docs/seminar-page-list.md` 부록 A). 참인 메커니즘을 대표 사례로 보여주는 것 — 그대로 제시해도 정당함. 실패 3종(필드 누락/키 드리프트/preamble)은 벤더 공식 문서가 인정하는 실제 현상. 데모 1·데모 3에도 "구성된 예시 장면" 라벨 명시(데모 3의 이슈 ID는 가상).
- 구성된 예시를 **"특정 harness를 실제 측정한 수치"인 것처럼 라벨링하지 말 것.** 속도·퍼센트 등 성능 숫자 금지(품질·재현성으로만).
- 재현성 주장은 "스키마 일치/같은 구조"까지만 — "매번 동일한 JSON" 같은 값 결정론 표현 금지 (재발 주의).
- 사내 도구(15장)·판단 사례(16장) 세부는 사용자 확인 전까지 일반적 서술 유지 — 사내 서비스 실제 사양을 지어내지 말 것. 데모 1 대화의 디테일(로그 3파일·12건 등)도 illustrative — 팀 용어와 어긋나면 교체. 용어는 팀 실사용 기준(예: "회귀"가 아니라 영문 "regression").
- 사용자는 대화로 얼개를 잡아가는 방식을 선호. 큰 결정은 먼저 제안하고 확인받기.
