- 최종 산출물 형태: 가로 슬라이드 HTML, 키보드 네비게이션, 애니메이션·인터랙션.
- 문서는 개념 수준이며 버전 표기(vX.Y)로 관리. 갱신 시 버전과 아래 상태를 함께 업데이트.

## 전체 흐름 (확정)

```
도입    AI agent란 게 있다 (+ 오프닝 데모 Ex1: 이 자료를 만든 대화 과정 애니메이션)
본문 1  사내/사외 정보 흐름   — 우리 환경부터
본문 2  plugin / skill / mcp — 부품을 이해한다 (+ Ex2)
본문 3  harness              — 틀로 감싸 제대로 쓴다 (+ Ex4)
마무리  (보류)
```

## 확정된 결정

- 핵심 서사: AI agent가 있다 → 이런 장점이 있다 → 이렇게 쓰면 된다 (하나의 큰 맥락, 첫 소개)
- 청자는 AI agent 이해도가 낮음 → 개념은 도식·비유 중심으로 쉽게
- 본문 3파트 구조, 난이도 우상향(환경 → 부품 → 틀)
- 오프닝 데모(Ex1)는 도입 "정의·패러다임" 자리에 배치 = 개념의 증명. 자동재생, 대화 재생 + 얼개 성장(A안)
- 데모는 라이브가 아니라 **미리 만든 예시**를 자료에 삽입
- Ex2·Ex4는 **하나의 "문서 parsing" 작업을 공유** (Ex2=기본 동작, Ex4=같은 작업의 harness 유무 대조)
- Ex4 대조 축은 **품질 + 재현성** (속도·퍼센트 등 성능 수치는 쓰지 않음)
- GitOps 협업 소재는 제외
- 설치 주의사항은 본문 2에 흡수, 간단히만

## 아직 안 정한 것 (다음 결정 대상)

- **본문 2의 "유용한 것들"(대표 plugin/skill/mcp) 페이지 수** — (a) 1장에 추림 / (b) 카테고리별 3장 / (c) 본문 1~2개 + 나머지 부록. 이것만 정하면 페이지 골격 확정.
- 마무리 파트 전체 방향

## 작업 시 주의 (가드레일)

- **Ex4 예시는 "구성된 illustrative 사례"로 이미 채워져 있음** (`docs/seminar-page-list.md` 부록 A). 이는 참인 메커니즘(스키마 검증 + 재시도가 품질·재현성을 높임)을 대표 사례로 보여주는 것으로, 입문 세미나에서 **그대로 illustrative로 제시해도 정당함**. 사용자가 원하면 자신의 harness 실제 실행 결과로 교체 가능(선택 사항, 필수 아님).
- 다만 이 구성된 예시를 **"특정 harness를 실제 측정한 수치"인 것처럼 라벨링하지 말 것.** 속도·퍼센트 등 성능 숫자도 쓰지 않기로 함(품질·재현성으로만).
- 시각 표현·디자인·애니메이션 구체화는 아직 보류 상태. 사용자가 넘어가자고 하기 전엔 개념 수준 유지.
- 사용자는 한 번에 다 만들기보다 대화로 얼개를 잡아가는 방식을 선호. 큰 결정은 먼저 제안하고 확인받기.
