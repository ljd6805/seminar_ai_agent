도입 · 정의

AI agent란?

방금 보신 것처럼 — 목표를 주면 필요한 도구를 스스로 선택하고, 여러 단계를 거쳐 작업을 완수하는 AI입니다. 한 번의 질문·답변이 아니라, 결과를 확인하고 고치며 끝까지 진행합니다.

기존 방식

내가 한 단계씩

파일 열고 · 찾고 · 고치고 — 매 단계를 사람이 지시.

agent 방식

의도를 말하면

"이걸 이렇게 해줘" → 필요한 도구를 스스로 선택해 실행하고, 결과를 확인·수정.

핵심 세 가지 — 도구 사용 · 다단계 진행 · 결과 확인

도입 · 왜 지금

왜 지금, 우리가

모델이 외부 도구를 다루게 된 2023, 연결이 표준이 된 2024 — agent가 실무에 들어올 조건은 최근에야 갖춰졌습니다. 그리고 우리에겐 이미 사내 LLM이 있습니다.

지금까지

반복 작업에 시간을

문서 정리, 로그에서 필요한 줄 골라내기, 형식 맞추기 — 매번 사람 손으로. 정작 판단·설계에 쓸 시간이 밀려납니다.

agent를 쓰면

판단·설계에 시간을

반복은 agent에게 넘기고, 사람은 "무엇을 시킬지, 무엇을 믿을지"에 집중합니다.

이미 쓸 수 있는 환경이 회사 안에 있습니다 — 먼저 우리 환경부터 봅니다.

본문 1 · 우리 환경부터

우리 환경 — 정보는 어디로 가나

쓸 수 있는 경로는 둘 — 구분 기준은 하나, 데이터가 어디로 가는가입니다.

사내에 머문다

사내 서버에서 서빙되는 LLM — API로 직접 호출하거나 OpenCode 같은 에이전트 클라이언트로 사용. 설계·검증 데이터가 회사 경계를 벗어나지 않습니다.

사내 APIOpenCode지금 사용 가능
경계를 넘는다

사외 Claude Code(도입 예정) — 입력한 코드·데이터가 회사 밖 서버로 전송됩니다. 무엇을 보내는지 알고 써야 합니다.

외부 전송도입 예정

원칙: 민감·설계 데이터는 사내에서. 판단이 서지 않으면 보안 검토 먼저.

본문 2 · 확장 요소를 이해한다

확장 요소 셋 — skill · MCP · plugin

skill

모델이 불러 쓰는 전문 절차

재사용할 절차·전문지식을 SKILL.md로 정의 — 작업과 맞으면 모델이 스스로 불러와 따릅니다.

MCP

외부 연결 표준 프로토콜

AI를 외부 도구·데이터에 잇는 개방형 규격(client–server). 서버가 tools·resources·prompts를 노출 — 공식 문서의 비유로 "AI용 USB-C".

plugin

확장 배포 패키지

skill·MCP·hook(정해진 시점의 자동 동작)을 묶어 버전관리·설치하는 꾸러미 — marketplace에서 설치.

본문 2 · 동작 원리

어떻게 동작하나

skill · 로드

필요할 때만 로드

① 평소엔 이름·설명만 상주 → ② 작업과 맞을 때 본문·스크립트 로드 — 필요할 때만 펼치기(progressive disclosure)로 컨텍스트 절약.

MCP · 연결

연결하고 골라 호출

① 서버 연결·tool 목록 교환 → ② 모델이 필요한 tool을 골라 호출·결과 수신. 로컬·원격 모두 연결.

plugin · 배포

묶어서 설치

① marketplace 등록 → ② 설치하면 안에 담긴 skill·명령이 plugin명:이름 꼬리표(이름 충돌 방지)로 자동 등록.

관계: plugin은 담아 배포하는 꾸러미 — skill·MCP는 그 안에 담기는 능력·연결(단독 사용도 가능). 다음 장에서 skill이 실제로 동작하는 것을 봅니다.

본문 2 · 데모 2 · 기본 동작

눈으로 보기 — 문서 한 장을 구조화

비정형 입력 · 회의 메모

전부 외울 필요는 없습니다. 이런 종류의 도구들이 있다는 지도만 — 종류마다 예 하나씩.

주의는 하나 — 일부 MCP·plugin은 외부로 연결됩니다. 설계 자산이 나갈 수 있으니 애매하면 보안 먼저(본문 1의 원칙). — 이제 이 확장 요소들을 틀로 감싸 제대로 쓰는 법으로 갑니다.

본문 3 · 틀로 감싸 제대로

harness — 모델을 감싸는 실행 틀

확장 요소를 정해진 순서·규칙으로 묶고, 출력을 검증하고, 실패하면 재시도하는 실행 틀. — DUT를 감싸 정해진 자극·검사를 가하는 verification harness와 같은 구조, 대상이 모델일 뿐입니다.

확장 요소+순서·규칙+검증·재시도=검증되고 재현되는 실행

공식 용어입니다 — Claude Code 문서는 스스로를 "agentic harness", 모델에 도구·컨텍스트·실행 환경을 붙이는 틀이라고 설명합니다.

본문 3 · 장점과 단점

무엇을 얻고, 무엇을 치르나

얻는 것

일관된 품질 · 재현성

품질 — 정의한 검증을 통과한 결과만 출력. 재현성 — 몇 번을 실행해도 동일한 구조.

치르는 것

초기 설계 · 유지보수

틀을 처음 설계하는 비용, 그리고 작업이 바뀌면 틀도 손봐야 하는 관리 비용.

그래서 다음 장 — 같은 작업을 harness가 있을 때와 없을 때로 나란히 봅니다.
