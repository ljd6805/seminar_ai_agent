AI agent는 어디서 왔나

규칙을 사람이 다 짜던 시절부터 스스로 도구를 골라 일하는 단계까지 — 각 단계가 다음 단계의 발판이 됩니다.

2012AlexNet — 규칙 대신 데이터로 학습이 통한 첫 순간 (GPU)

2017Transformer — 문맥을 통째로 보는 구조(self-attention)로 대규모 학습 — LLM의 뼈대

2020GPT-3 — 예시 몇 개만으로 재학습 없이(few-shot) — 프롬프트로 지시

2022ChatGPT — 사람 피드백(RLHF)으로 지시를 잘 따름 — 대화로 대중화

2023function calling — 모델이 외부 도구를 호출(표준화) — 말에서 행동으로

2024–25MCP · agent — 도구 연결의 표준(MCP) + 스스로 도는 루프 = AI agent

도입 · 정의

AI agent란?

방금 보신 것처럼 — 목표를 주면 필요한 도구를 스스로 선택하고, 여러 단계를 거쳐 작업을 완수하는 AI입니다. 한 번의 질문·답변이 아니라, 결과를 확인하고 고치며 끝까지 진행합니다.

기존 방식

내가 한 단계씩

파일 열고 · 찾고 · 고치고 — 매 단계를 사람이 지시.

agent 방식

의도를 말하면

"이걸 이렇게 해줘" → 필요한 도구를 스스로 선택해 실행하고, 결과를 확인·수정.

핵심 세 가지 — 도구 사용 · 다단계 진행 · 결과 확인

도입 · 왜 지금

왜 지금, 우리가

모델이 외부 도구를 다루게 된 2023, 연결이 표준이 된 2024 — agent가 실무에 들어올 조건은 최근에야 갖춰졌습니다. 그리고 우리에겐 이미 사내 LLM이 있습니다.

지금까지

반복 작업에 시간을

문서 정리, 로그에서 필요한 줄 골라내기, 형식 맞추기 — 매번 사람 손으로. 정작 판단·설계에 쓸 시간이 밀려납니다.

agent를 쓰면

판단·설계에 시간을

반복은 agent에게 넘기고, 사람은 "무엇을 시킬지, 무엇을 믿을지"에 집중합니다.

이미 쓸 수 있는 환경이 회사 안에 있습니다 — 먼저 우리 환경부터 봅니다.

본문 2 · 확장 요소 ① skill

skill — 모델이 불러 쓰는 전문 절차

agent를 확장하는 세 가지 — skill·MCP·plugin. 그 첫 번째, skill은 재사용할 절차·전문지식을 파일로 정리해 둔 것입니다. 반복 작업의 노하우를 한 번 적어 두면, 모델이 관련 작업마다 같은 절차를 그대로 따릅니다.

구조 — 폴더 하나 = skill 하나
review-spec/ — skill 폴더
review-spec/ ├─ SKILL.md ← 이름·설명 + 절차 본문 └─ scripts/ … ← 참조 자료·스크립트(선택)

SKILL.md 하나면 시작 — 잘 쓰는 사람의 절차가 팀의 절차가 됩니다.

동작 원리 — 필요할 때만 펼치기 (progressive disclosure)
① 평소
이름·설명만 상주 — 컨텍스트를 거의 쓰지 않음
② 작업과 맞으면
SKILL.md 본문을 불러와 절차를 따름
③ 더 필요하면
본문이 가리키는 참조 파일·스크립트를 그때만 로드

본문 2 · 확장 요소 ② MCP

MCP — 외부 연결 표준 프로토콜

두 번째, MCP(Model Context Protocol)는 AI를 외부 도구·데이터에 잇는 개방형 표준입니다. 도구마다 제각각이던 연결 방식을 하나로 — 공식 문서의 비유로 "AI용 USB-C".

구조 — client · server
AI 클라이언트모델이 있는 쪽
⇄
MCP 서버도구·데이터가 있는 쪽

서버가 노출하는 세 가지:

tools · 도구resources · 데이터prompts · 워크플로
동작 원리 — 연결하고, 골라 호출
① 연결
서버에 접속, 쓸 수 있는 tool 목록을 교환
② 선택·호출
모델이 필요한 tool을 골라 호출 — 로컬·원격 모두
③ 결과 수신
응답을 받아 다음 판단에 사용

개방형 표준이라 한 번 만든 서버는 어떤 AI 클라이언트에서든 — 발표사(Anthropic) 외에 OpenAI·Google도 채택한 공용 규격입니다.

본문 2 · 확장 요소 ③ plugin

plugin — 확장 배포 패키지

세 번째, plugin은 skill·MCP·hook(정해진 시점의 자동 동작) 등을 한 꾸러미로 묶은 배포 단위입니다. 잘 만든 조합을 하나씩 깔지 않고 한 번에 설치·버전관리 — 팀에 공유하는 그릇입니다.

구조 — 담아서 배포하는 꾸러미
my-plugin/ — plugin 폴더
my-plugin/ ├─ plugin.json ← 이름·버전 (이름이 곧 꼬리표) ├─ skills/ hooks/ ← 담긴 확장 요소들 └─ .mcp.json ← MCP 서버 연결
설치와 이름표

marketplace에서 한 번에

설치하면 안에 담긴 skill·명령이 plugin명:이름 꼬리표(이름 충돌 방지)로 자동 등록됩니다.

관계: plugin은 담아 배포하는 꾸러미 — skill·MCP는 그 안에 담기는 능력·연결(단독 사용도 가능). 다음 장에서 skill이 실제로 동작하는 것을 봅니다.

본문 2 · 데모 2 · 기본 동작

눈으로 보기 — 문서 한 장을 구조화

meeting-memo.txt — 비정형 입력
10/24 주간 동기화. 참석: 지훈, 민아, 세영. - 로그인 API 지연, 원인은 토큰 갱신. 지훈이 금요일까지 수정. - 대시보드 3안 중 2안 결정. 민아 다음 주 월요일까지. - 다음 회의 10/31 오후 3시.
output.json — skill이 구조화한 출력
"title": "주간 동기화", "date": "2024-10-24",
"participants": ["지훈","민아","세영"],
"decisions": ["대시보드 시안 2안으로 결정"],
"action_items": [
{ "owner":"지훈", "task":"토큰 갱신 수정", "due":"2024-10-25" },
{ "owner":"민아", "task":"대시보드 2안 반영", "due":"2024-10-28" } ],
"next_meeting": "2024-10-31 15:00"

핵심: 비정형 입력 → 구조화 출력. 이 변환 규칙을 담은 것이 skill — 설계·검증에선 이 자리에 스펙 문서·검증 로그가 들어갑니다.

본문 2 · 골라 쓰기 · 설치

전부 외울 필요는 없습니다. 이런 종류의 도구들이 있다는 지도만 — 종류마다 예 하나씩.

주의는 하나 — 일부 MCP·plugin은 외부로 연결됩니다. 설계 자산이 나갈 수 있으니 애매하면 보안 먼저(본문 1의 원칙). — 이제 이 확장 요소들을 틀로 감싸 제대로 쓰는 법으로 갑니다.

본문 3 · 틀로 감싸 제대로

harness — 모델을 감싸는 실행 틀

확장 요소를 정해진 순서·규칙으로 묶고, 출력을 검증하고, 실패하면 재시도하는 실행 틀. — DUT를 감싸 정해진 자극·검사를 가하는 verification harness와 같은 구조, 대상이 모델일 뿐입니다.

확장 요소
+
순서·규칙
+
검증·재시도
=
검증되고 재현되는 실행
공식 용어입니다 — Claude Code 문서는 스스로를 "agentic harness", 모델에 도구·컨텍스트·실행 환경을 붙이는 틀이라고 설명합니다.

본문 3 · 장점과 단점

무엇을 얻고, 무엇을 치르나

얻는 것

일관된 품질 · 재현성

품질 — 정의한 검증을 통과한 결과만 출력. 재현성 — 몇 번을 실행해도 동일한 구조.

치르는 것

초기 설계 · 유지보수

틀을 처음 설계하는 비용, 그리고 작업이 바뀌면 틀도 손봐야 하는 관리 비용.

그래서 다음 장 — 같은 작업을 harness가 있을 때와 없을 때로 나란히 봅니다.
