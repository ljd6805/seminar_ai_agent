도입 · 정의

AI agent란?

목표를 주면, 스스로 도구를 골라 여러 단계를 밟아 일을 끝내는 AI. 한 번의 질문·답변이 아니라, 확인하고 고치며 끝까지 진행합니다.

기존 방식

내가 한 단계씩

파일 열고 · 찾고 · 고치고 — 매 단계를 사람이 지시.

agent 방식

의도를 말하면

"이걸 이렇게 해줘" → 알아서 도구를 쓰고 결과를 확인·수정.

핵심 3가지

도구·다단계·검증

도구를 쓴다 / 여러 단계를 스스로 / 결과를 확인하고 고친다.

본문 1 · 데이터 경계

정보는 어디로 가나

사내에 머문다

사내 LLM(API·OpenCode). 설계·검증 데이터가 회사 경계를 벗어나지 않음.

경계를 넘는다

사외 Claude Code. 보낸 내용이 외부로 나감 — 무엇을 보내는지 통제 필요.

원칙: 민감·설계 데이터 → 사내. 공개·비민감 실험 → 사외. 애매하면 보안 먼저 확인.

본문 2 · 부품을 이해한다

부품 셋 — plugin · skill · mcp

skill

전문 기술

특정 작업을 잘 해내는 방법 묶음. "문서를 구조화해라" 같은 능력.

mcp

표준 커넥터

외부 시스템·데이터에 연결하는 규격. agent를 우리 도구에 꽂는 어댑터.

plugin

패키지

위 것들을 묶어 배포·설치하는 단위. 골라 담는 부품 상자.

본문 2 · 동작 원리

어떻게 맞물리나 — 레고 비유

agent는 작업대, skill·mcp는 그 위에 꺼내 쓰는 블록. 필요할 때 맞는 블록을 집어 끼우고, 다 쓰면 내려놓습니다.

요청→agent가 필요한 skill/mcp를 고름→도구 실행→결과 확인→완료

부품을 아는 순간, 다음 장에서 실제로 움직이는 걸 눈으로 봅니다.

본문 3 · 틀로 감싸 제대로

harness — 부품을 감싸는 틀

앞의 부품(skill·mcp)을 정해진 순서와 규칙으로 묶어, 항상 같은 품질로 돌아가게 하는 틀. 블록에 조립 설명서와 작업대를 붙이는 셈.

부품+순서·규칙+검증=믿고 쓰는 자동화

본문 3 · 장점과 단점

무엇을 얻고, 무엇을 치르나

얻는 것

일관된 품질 · 재현성

검증을 통과한 것만 나옴 → 항상 유효. 몇 번을 돌려도 같은 형태. 자동 검증.

치르는 것

초기 설계 · 유지보수

틀을 짜는 첫 수고와, 바뀌면 손봐야 하는 관리 비용.

그래서 다음 장 — 같은 작업을, 틀이 있고 없고로 나란히 봅니다.
