# 발표자 노트 — AI agent 입문 세미나 (53장 · 90분) v1.0

각 항목 구조: **한 문장**(이 장의 핵심) / **풀어서**(말로 할 설명) / **우리 일로**(설계·검증 연결) / **주의**(오해·한계).
슬라이드 번호는 v1.7(53장) 기준.

## 발표 전체를 관통하는 문장 (고정)

> **AI agent는 사람의 판단을 대체하는 도구가 아니라, 판단 전에 필요한 반복 작업을 줄이는 도구다.**

네 기둥: ① agent = LLM + 도구 + 반복 루프 실행자 · ② skill = 일하는 방법 · ③ MCP = 일할 시스템 연결 · ④ harness = 다음 단계로 넘어가도 깨지지 않게 감싸기.

## 반드시 바로잡을 오해 6 (해당 장에서 명시적으로 언급)

1. LLM은 항상 사실을 말하는 것이 아니다 → 5·8장
2. agent는 무제한으로 알아서 도는 것이 아니다 — 권한·한도·확인 안에서 → 11장
3. 사내 AI라도 접근 권한·프로젝트 경계는 지켜야 한다 → 18·20·21장
4. MCP는 연결 규격이지 보안 정책 자체가 아니다 → 33·35장
5. schema validation은 내용의 진실성을 보장하지 않는다 → 42장
6. harness는 모델을 믿기 위한 장치가 아니라 모델 사용을 통제하기 위한 장치다 → 41장

## 3장 · 오늘의 결론

- **한 문장**: 결론을 3분 안에 먼저 준다 — 반복은 agent, 판단은 사람.
- **풀어서**: "오늘 이 한 장만 기억하셔도 됩니다. 뒤의 48장은 이 네 줄의 각주입니다." 네 기둥을 천천히 한 번씩 읽어 준다.
- **우리 일로**: regression 로그 분류·회의록 action item·spec 변경점 요약 — 전부 '판단 전 준비' 작업.
- **주의**: 여기서 기술 용어 설명을 시작하지 말 것. "낯설어도 됩니다"로 넘어간다.

## 4장 · 역사

- **한 문장**: 뒤의 세 노드(function calling·MCP·agent loop)가 최근 2~3년 — "왜 지금"의 답.
- **풀어서**: 앞의 네 노드는 30초 안에 훑는다. 역사에 시간을 쓰지 않는다(총 1.5분 이내).
- **주의**: AlexNet은 "최초의 머신러닝"이 아니다 — 데이터 기반 학습은 그 전에도 있었고, GPU 딥러닝의 실용성을 증명한 '상징적 사건'이라고만. MCP도 "표준 확정"이 아니라 "공용 규격으로 확산 중".

## 5장 · LLM

- **한 문장**: LLM이 하는 일은 다음 말(엄밀히는 다음 토큰) 예측 하나다.
- **풀어서**: "입력된 문맥을 보고 다음 토큰을 확률적으로 예측 → 반복하면 문장·코드·표·요약이 나온다. 그래서 유창하지만, 유창하다고 사실인 건 아니다." 슬라이드 하단 각주를 소리 내어 읽는다: "엄밀히 말하면 '다음 말'이 아니라 '다음 토큰'을 예측합니다. 오늘은 이해를 돕기 위해 '다음 말'이라고 표현하겠습니다."
- **우리 일로**: 시뮬레이션 로그 요약이 매끄러워 보여도 근거 확인은 필요하다는 복선.
- **주의**: "AI가 이해한다/안다" 같은 표현을 쓰지 않는다 — 예측이라고만.

## 6장 · 컨텍스트

- **한 문장**: context window에는 한도가 있다 — 넘으면 잘리거나 요약되고, 길수록 관련 정보를 집어내기 어렵다.
- **풀어서**: "모델의 작업 기억" 비유는 유지하되, 사람처럼 기억이 흐려진다기보다 **입력 공간의 한계와 정보 선택의 문제**로 설명.
- **우리 일로**: "필요한 절차와 자료만 골라 넣는 것이 중요합니다. skill이 바로 그 정리법입니다" — skill 복선.

## 7장 · 토큰

- **한 문장**: token은 모델이 읽고 쓰는 기본 단위 — 길수록 비용·속도·컨텍스트를 먹는다.
- **우리 일로**: "수십 MB regression log 전체를 넣기보다, fail case 주변 로그와 summary만 먼저 넣는 것이 더 효과적입니다."
- **주의**: '많이 넣기'가 아니라 '잘 고르기'가 실력이라는 프레임.

## 8장 · 환각

- **한 문장**: 모델은 모른다고 멈추기보다 그럴듯한 답을 만드는 쪽으로 기울 수 있다 — 유창함은 신뢰성과 다르다.
- **풀어서**: "거짓말"이라고 부르지 않는다 — 의도가 아니라 예측 방식에서 오는 한계. 대응은 근거·도구·검증 절차로 감싸는 것(harness 복선).
- **우리 일로**: 근거 없는 root cause 서술이 제일 위험한 형태 — 그래서 triage skill에 "추정 금지"를 넣는다(29장 복선).

## 11장 · AI agent 정의

- **한 문장**: agent는 정해진 권한과 한도 안에서 이해→계획→실행→확인을 반복하는 실행자다.
- **풀어서**: loop를 단계로 읽어 준다 — ① 목표 이해 ② 계획 ③ 도구 호출/실행 ④ 결과 확인 ⑤ 부족하면 제한된 범위 안에서 재시도 ⑥ 최종 제출.
- **주의**: "될 때까지 알아서 돈다"처럼 들리지 않게 — 반드시 "권한·한도·확인"을 말에 넣는다. 마법이 아니라 통제 가능한 자동화 실행 흐름.

## 12장 · 챗봇 vs agent

- **한 문장**: 질문에 답하면 챗봇, 목표를 받아 도구를 쓰며 진행하면 agent에 가깝다.
- **주의**: agent가 항상 우월하다고 말하지 않는다 — 단순 질의응답엔 챗봇으로 충분. 여러 단계 + 도구 호출이 필요할 때 agent가 유리.

## 16장 · 무엇을 맡기나

- **한 문장**: AI가 잘하는 건 초안·분류·변환·요약 — 최종 판단은 사람.
- **우리 일로**: 맡기기 좋은 일(regression 실패 분류·spec 변경점 요약·review comment 정리·register table 변환·회의록 action item) vs 조심할 일(sign-off·architecture 결정·RTL 의미 변경·root cause 확정·coverage closure·보안/고객 커뮤니케이션)을 청중의 실제 업무명으로 읽는다.

## 18장 · 정보 경계

- **한 문장**: 구분 기준은 하나 — 데이터가 어디로 가는가. 그리고 사내라도 내 접근 권한 안에서만.
- **주의**: "사내면 다 된다"로 들리지 않게 — need-to-know는 사내 경로에서도 그대로.

## 20장 · 보내도 되는 것·안 되는 것

- **풀어서**: 실무 체크 한 줄을 소리 내어: "이 데이터를 외부 메일로 보내도 되는가? 아니오라면 사외 AI에도 넣으면 안 됩니다."
- **주의**: 슬라이드의 예시는 감 잡기용 — **정확한 기준은 팀 보안 규정이 우선**을 반드시 말한다.

## 21장 · 3문장 점검

- **한 문장**: 권한? 경계? 검증? — 세 번 다 "예"일 때 도구를 켠다.
- **풀어서**: 이 장은 청중이 사진 찍어 가는 장 — 천천히 하나씩.

## 22장 · OpenCode 설치

- **주의**: 설치 주소·사내 서버 설정은 사내 안내 페이지 참고로만 안내(화면 값은 placeholder).

## 24장 · 기본 모델의 한계 (두 공백)

- **한 문장**: 모델은 우리 절차를 모르고(공백 ①), 우리 시스템에 못 닿는다(공백 ②) — 절차는 skill, 연결은 MCP.

## 25–26장 · 데모 2 (업무 정의 → skill 저장·실행)

- **풀어서**: "정의할 수 있으면 맡길 수 있다"가 축. 26장 실행 셸에서 skill이 description 매칭으로 자동 로드되는 줄을 짚는다.
- **주의**: 구성된 예시 장면 라벨 언급. "매번 동일한 JSON"이 아니라 "같은 절차·같은 구조"로만 말한다.

## 27장 · skill 개념

- **한 문장**: skill은 매번 프롬프트를 길게 쓰지 않기 위한 팀 작업 매뉴얼 — 재사용 가능한 작업 설명서.
- **풀어서**: 단순 프롬프트 모음이 아니라 '언제 쓰나·기대 입력·작업 순서·출력 형식·하지 말 것·검증 방법'을 정하는 작업 지침.

## 29장 · regression-failure-triage skill

- **한 문장**: 회의 메모가 regression 로그로 바뀌었을 뿐 — 규격은 같다.
- **풀어서**: 절차 5번 "원인 확정이 어려우면 unknown으로 두고 추정하지 않는다"를 강조 — 환각(8장)을 절차로 누르는 장치.

## 30장 · 좋은/나쁜 skill

- **한 문장**: AI에게 맡기는 작업일수록 사람보다 더 명확한 작업 정의가 필요하다.

## 31–32장 · 데모 3 (MCP 없을 때/있을 때)

- **풀어서**: 없을 때 = 연결이 사람(로그인→검색→복사→붙여넣기, 매주 반복). 있을 때 = 같은 한 줄 부탁, 조회는 tool 호출.
- **주의**: 자동화된 건 **조회까지** — 우선순위 판단·의사결정은 사람이 확인. 이슈 ID는 가상(구성된 예시).

## 33장 · MCP 개념

- **한 문장**: MCP는 AI application과 외부 시스템을 잇는 공용 규격 — agent에게 손을 달아주는 것이지 판단력을 보장하는 것이 아니다.
- **주의**: "어디서나 꽂아 쓴다"가 아니라 "지원하는 클라이언트에서는 재사용이 쉬워진다". MCP는 연결 규격이지 보안 정책이 아니다 — 인증·권한·네트워크는 별도 설계(34장 caption).

## 35장 · MCP 통제

- **한 문장**: agent는 권한을 가진 자동화 실행기 — 권한 설계가 곧 안전 설계.
- **풀어서**: 표에서 prompt injection만 한 번 풀어 준다: "문서나 이슈 내용 자체가 agent에게 지시처럼 작동할 수 있다 — 그래서 tool 결과는 불신 데이터로 취급."

## 39–40장 · harness란 / UVM 나란히

- **한 문장**: harness는 형식과 실행을 통제하는 틀 — 구조는 verification harness와 같고 대상만 칩→모델.
- **풀어서**: sequence↔입력·절차, DUT↔모델, assertion↔스키마. **양쪽 다 최종 판단은 사람**: spec 해석은 엔지니어가, 의미·업무 판단은 사람이.
- **주의**: "믿고 쓸 수 있게"가 아니라 "깨지지 않게" — 내용의 진실성 보장이 아니다.

## 41장 · harness 구성 요소

- **한 문장**: harness는 스키마 검증 하나가 아니라 통제 장치들의 묶음 — 모델을 믿기 위한 장치가 아니라 모델 사용을 통제하기 위한 장치.

## 42장 · 스키마 게이트

- **한 문장**: 스키마는 모양을 검증한다 — JSON이 맞는지 보는 것이지, 내용이 참인지 보는 것이 아니다.
- **풀어서**: 장점(형식 일정·누락 필드 검출·파싱 실패 감소·retry 조건 명확)과 한계(내용 사실성 보장 없음·근거 없는 내용도 올바른 JSON일 수 있음)를 모두.
- **우리 일로**: "assertion이 interface violation을 잡아도 spec 자체가 맞는지는 별도 리뷰가 필요한 것과 같습니다."

## 43장 · 장단점 (재현성)

- **주의**: 재현성 = "매번 같은 답"이 아니라 "매번 같은 검증 절차와 같은 출력 구조를 통과". LLM 출력은 확률적일 수 있으므로 구조·검증·로그·승인 절차가 중요.

## 45장 · 자동화 레벨 L0–L5

- **한 문장**: 처음부터 L5로 가면 사고도 자동화된다 — 시작은 L1~L2.
- **풀어서**: 오늘의 데모가 사다리에 그대로 놓임(데모 1=L1, 데모 2=L2, 데모 3=L3, 데모 4=L4). L5는 human approval이 전제.

## 46장 · 데모 4

- **주의**: 대조 축은 품질+재현성(스키마 일치/같은 구조까지만) — 성능 수치·퍼센트 금지. bad run 3종(필드 누락·키 드리프트·preamble)은 벤더 문서가 인정하는 실제 현상.

## 49장 · 좋은/나쁜 요청

- **한 문장**: 작게 · 구체적으로 · 검증 가능하게.
- **풀어서**: 좋은 요청 3개를 그대로 읽어 준다 — 청중이 복사해 갈 문장.

## 50장 · 첫걸음 CTA

- **한 문장**: 첫 목표는 자동화가 아니라, 내가 검증 가능한 반복 작업 하나를 줄이는 것.
- **풀어서**: "이번 주에 딱 하나 — 사내에서, 검증 가능한 것으로." 화면의 두 프롬프트 중 하나를 그대로 써도 된다고 안내.
