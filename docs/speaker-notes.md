# 발표자 노트 — AI Agent 입문 세미나 (57장 · 90분) v1.9

각 항목 구조: **한 문장**(이 장의 핵심) / **풀어서**(말로 할 설명) / **우리 일로**(설계·검증 연결) / **주의**(오해·한계).
슬라이드 번호는 v2.4(57장 = 본편 40 + 부록 17) 기준 — 구 20장(Skill 구조·동작 READ)이 20 구조 / 21 실행 과정 / 22 만드는 법 세 CORE 장으로 분리돼 구 21장 이후 번호가 2씩 밀림. 발표는 `?mode=presentation`(CORE+DEMO 38장) 링크 사용 가능. 부록(41~57장)은 발표 시 넘기지 않는다. v1.8.1부터 슬라이드 하단은 결론·주의만 담는다 — **장 사이 전환 멘트는 발표자가 말로**(각 장 "풀어서" 참고).

## 발표 전체를 관통하는 것 (고정)

> **AI Agent는 사람의 판단을 대체하는 도구가 아니라, 판단 전에 필요한 반복 작업을 줄이는 도구다.**

- **관통 시나리오**: Regression Triage Agent 하나를 — 데모 1(prompt만) → 데모 2(Skill) → 데모 3(MCP) → 데모 4(Harness) → 사람 승인으로 키운다. "다음 단계로 넘어갈 때마다 무엇이 부족했는지"가 서사의 엔진.
- **4요소(3장, Master Visual 안 한 문장 정의 — v1.8)**: Agent(목표를 향해 도구를 사용하며 작업을 이어가는 실행 시스템) · Skill(반복 업무의 절차와 기준을 담은 작업 매뉴얼) · MCP(Agent와 데이터·도구를 연결하는 공용 규격) · Harness(권한·검증·로그·승인으로 실행을 통제하는 구조) — 종류로 구분해 기억: 실행 시스템 / 작업 매뉴얼 / 공용 규격 / 통제 구조.
- **7~10장 질문 체계(v2.3)**: 7 WHAT(무엇이 다른가 — 실행 구조) → 8 WHY(왜 쓰나 — 반복에 묶인 시간을 판단으로) → 9 WHO(무엇을 나눠 맡나 — 사람→Agent→사람) → 10 OPERATE(어떻게 주고받나 — 검증 루프). 그림 소유: 레일·시간 이동=8, 사람→Agent→사람 구조=9, 반복·수정 방법=10 — 장끼리 그림을 반복하지 않는다.
- **4원칙(10장)**: 작게 시작 · 출력 구조화 · 근거 남기기 · 사람 승인 — 이후 배지가 보일 때마다 한 번씩 짚는다.

## 반드시 바로잡을 오해 6 (해당 장에서 명시적으로 언급)

1. LLM의 유창함이 사실성을 보장하지 않는다 → 본편 7장(하단 경고) · 상세는 부록 41·44장

3. 사내 AI라도 접근 권한·프로젝트 경계는 지켜야 한다 → 12·14장

5. schema validation은 내용의 진실성을 보장하지 않는다 → 32장
6. Harness는 모델을 믿기 위한 장치가 아니라 모델 사용을 통제하기 위한 장치다 → 31장

## 3장 · 오늘의 결론 · 4요소

- **한 문장**: 결론을 3분 안에 — 반복은 Agent, 판단은 사람. 그리고 오늘의 네 단어를 한 문장씩만 짚는다(20~30초).
- **풀어서(그림의 네 정의를 그대로)**: "오늘 네 단어가 나옵니다. Agent는 실제 작업을 이어가는 실행 시스템이고, Skill은 반복 업무의 절차와 기준을 담은 작업 매뉴얼입니다. MCP는 Agent를 데이터와 도구에 연결하는 공용 규격이고, Harness는 그 실행을 권한과 검증, 승인으로 통제하는 구조입니다. 지금 외우실 필요는 없습니다. 같은 작업을 키워 가면서 하나씩 다시 보겠습니다."
- **주의**: 이 장에서 MCP 구조나 Skill 파일 형식을 상세히 설명하지 않는다. 네 요소를 모두 같은 종류의 표준이나 제품처럼 설명하지 않는다 — 실행 시스템 / 작업 매뉴얼 / 공용 규격 / 통제 구조로 종류가 다르다. 정의를 짚은 뒤 바로 4장 데모로 넘어간다.

## 4장 · 데모 1 (prompt + 제공된 로컬 로그)

- **한 문장**: 반복 실행된 테스트의 실패를 증상·로그 패턴으로 1차 분류 — 검증팀 용어로 Regression Triage(여기서 한 번만 설명).
- **권장 멘트**: "반복 실행된 테스트에서 나온 실패를 비슷한 증상과 로그 패턴으로 묶어 보겠습니다. 검증팀에서는 이런 작업을 Regression Triage라고 부릅니다." / "여기서 timeout이나 assertion은 원인 확정이 아니라 1차 분류 기준입니다."
- **구분해 말할 것**: 파일은 로그 3개, 처리 대상은 실패 12건. 저장된 Skill·MCP 연결 없이 제공된 로그와 기본 파일 도구만 사용(순수 대화만이 아님).
- **주의**: 구성된 예시 라벨 언급. 성공 데모가 아니라 "꽤 되지만 완전하지 않은 장면" — "중요한 것은 9건을 정리했다는 것만이 아니라, 근거가 부족한 3건을 사람에게 남겼다는 점입니다."

## 5장 · 데모 1 해부

- **한 문장**: 방금 Agent는 요청을 작업으로 풀고(목표 구체화), 제공된 로그를 도구로 처리하고(도구로 실행), 사람의 피드백을 받아 결과를 보완했다(피드백 반영).
- **주의**: 4장에서 관찰된 행동만 말한다 — Agent가 모든 계획을 자율적으로 세운 것도, 자기 결과의 진실성을 스스로 보장한 것도 아니다. 결과 보완은 사람 피드백의 반영. 근거 부족 3건은 여전히 사람 판단으로 남았다.
- **6장 브리지(구두)**: "목표를 작업으로 풀고, 도구로 실행하고, 결과를 확인해 보완하는 이 반복을 하나로 묶으면 AI Agent의 정의가 됩니다."

## 6장 · AI Agent 정의

- **한 문장**: Agent는 단순히 대답하는 모델이 아니라, 목표를 받아 다음 행동을 결정하고, 도구 실행 결과를 관찰하며 작업을 이어 가는 실행 시스템이다.
- **연결**: 4장이 사례, 5장이 관찰, 6장이 일반화 — 목표를 작업으로 풀고, 다음 행동을 결정하고, 도구를 실행하고, 그 결과를 관찰해 다음 행동을 이어 간다.
- **경계**: 행동 결정→도구 실행→결과 관찰은 Agent의 내부 실행 루프. 허용 행동·도구 권한·출력 검증·재시도 한도·중단·사람 이관은 Harness가 다룬다(29–34장).
- **주의**: 결과 관찰은 정확성 검증이나 사실성 보장을 뜻하지 않는다. 작업물의 근거 확인과 최종 판단·승인 책임은 사람에게 있다. (오해 ②)

## 7장 · LLM은 모델, AI Agent는 실행 시스템

- **한 문장**: LLM은 언어를 생성하는 모델이고, AI Agent는 그 LLM에 허용된 도구와 행동–관찰 루프를 결합해 작업을 이어 가게 만든 실행 시스템이다.
- **풀어서**: "채팅창이냐 터미널이냐가 기준은 아닙니다. 챗봇은 기술적 반대 개념이 아니라 인터페이스·제품 형태입니다. LLM은 모델이고, Agent는 그 LLM을 포함한 실행 시스템입니다." 같은 요청("자동 테스트 실패를 정리해줘")을 두 구조에 넣은 좌우 그림을 그대로 읽는다 — 왼쪽은 입력된 문맥에서 응답 생성, 오른쪽은 LLM에 허용된 도구와 행동–관찰 루프(행동 결정→도구 실행→결과 관찰)가 결합돼 작업 결과가 남는다.
- **포함 관계**: "Agent 안에도 같은 역할의 LLM이 들어 있습니다. Agent가 별도의 더 똑똑한 모델이라는 뜻은 아닙니다." — 좌우의 LLM 노드가 같은 모양·같은 색인 것을 손으로 가리킨다. 차이는 도구·상태·행동–관찰 루프.
- **선 긋기**: "단순한 질문과 답변은 LLM이나 챗봇으로 충분합니다. 파일을 읽고, 도구를 사용하고, 도구 실행 결과를 관찰하며 작업을 이어 가야 할 때 Agent 구조가 필요합니다."
- **주의**: 하단 경고를 반드시 입으로 — "도구와 반복이 추가돼도 결과가 자동으로 참이 되는 것은 아닙니다." (오해 ① — 상세는 부록 41·44장). Agent가 도구 실행 결과를 관찰하고 다음 행동을 결정할 수 있다는 것이지, 정확성을 스스로 보장한다는 뜻이 아니다.
- **8장 브리지(구두)**: "Agent가 응답 모델을 넘어 실행 시스템이 되면서, 사람이 직접 하던 중간 반복 일부를 맡길 수 있게 됐습니다. 그 시간이 어디로 가는지부터 보겠습니다."

## 8장 · 왜 쓰나 — 반복에 묶인 시간을 판단으로 (WHY)

- **한 문장**: Agent가 더 똑똑해서가 아니라, 반복 실행을 맡겨 사람의 시간을 목표·기준과 검증·판단으로 옮기기 위해 쓴다.
- **풀어서**: 위 레일(사람이 여섯 단계 전부 — 점선 세 단계가 반복 실행 구간)을 읽고, "반복 실행 구간이 판단·설계의 시간을 밀어냅니다"를 그대로 소리 내어. 이어서 시간의 이동 두 줄 — "반복 실행은 Agent가 권한·한도 안에서, 사람의 시간은 무엇을 시킬지와 무엇을 믿을지로." 3장 핵심 메시지("판단을 대체하는 게 아니라 판단 전 반복을 줄인다")가 업무 흐름에서 뜻하는 것이 바로 이 그림.
- **주의**: 사람 대체·"처음과 끝에만 있는 구조"로 설명하지 않는다(중간 개입은 9장 하단 한 줄). 모든 반복 업무가 적합하다고 말하지 않는다. 시간 절감 수치를 말하지 않는다.
- **9장 브리지(구두)**: "시간을 옮기려면, 이제 누가 무엇을 맡을지 정해야 합니다."

## 9장 · 역할 분담 — 사람 · Agent · 사람 (구 무엇을 맡기나)

- **한 문장**: 사람은 목표·기준·검증·판단을 맡고, Agent는 범위가 제한되고 결과를 확인할 수 있는 반복 실행을 맡는다.
- **풀어서**: 세 카드(사람: 목표·기준·범위 / AI Agent: 탐색·분류·변환·초안 / 사람: 근거 확인·수정·판단·승인)를 왼쪽부터. 하단 한 줄 — 사람은 필요할 때 중간 결과를 확인하고 방향을 수정한다(수정을 주고받는 방법 자체는 10장 검증 루프가 소유).
- **우리 일로**: 하단 스트립 — 맡기기 좋은 중간 작업(자동 테스트 실패 1차 분류·spec 변경점 요약·형식 변환·초안) vs 사람에게 남길 판단(sign-off·architecture 결정·RTL 의미 변경·root cause 확정)을 실제 업무명으로 읽는다.
- **주의**: 사람은 필요할 때 중간에도 개입한다. Agent가 판단 책임을 가져가는 게 아니다. 사람→Agent→사람은 단순화된 기본 모델이다(상세는 부록 48장).
- **10장 브리지(구두)**: "역할을 나눴다면, 이제 중요한 것은 결과를 어떤 방식으로 주고받을지입니다."

## 10장 · Agent 도입 4원칙 — 작은 검증 루프

- **한 문장**: 좋은 Agent 협업은 한 번의 큰 위임이 아니라, 작은 결과를 구조화해서 받고, 근거를 확인한 뒤, 사람이 수정 또는 승인을 결정하는 반복 과정이다.
- **정식 문구(문서·배지 기준)**: ① 작게 시작한다 ② 출력을 구조화한다 ③ 근거를 남긴다 ④ 사람이 승인한다. 화면의 "근거 확인"은 "근거를 남긴다"의 협업 행동 표현 — 근거를 남긴다 = 사람이 실제 자료와 대조할 수 있게 한다.
- **풀어서**: 루프를 시계 방향으로 — 작게 시작(업무 범위 제한) → 출력 구조화(결과를 비교하기 쉽게) → 근거 확인(실제 로그·문서·코드와 대조) → 사람 승인(반영 여부와 다음 반복 결정). 그리고 분기 — "사람 승인은 루프의 끝이 아니라 분기점입니다. 수정이 필요하면 같은 범위에서 다시 돌리고, 승인되면 결과를 반영한 뒤 다음 작은 범위로 넘어갑니다." 업무 전체를 한 번에 자동화하지 않는다.
- **주의**: 4원칙이 정확성·안전을 보장하는 것이 아니다. Agent가 무한히 자동 반복하는 것이 아니다 — 사람 승인 없이 다음 범위로 넘어가지 않는다. 구조화된 출력이 내용의 진실성을 보장하지 않는다. 근거가 있다는 것과 근거 해석이 맞다는 것은 다르다. 매 반복은 같은 결과의 재생성이 아니라 기준 보완·형식 정리·누락 수정·범위 확대 중 하나가 달라진다.
- **연출**: 진입 시 루프가 ①→②→③→④ 순서로 한 바퀴만 그려진 뒤 분기(수정→확대)와 중앙 문구가 나타난다(~5초, 1회 재생 — 저절로 다시 돌지 않음). 애니메이션이 도는 동안 lead 문장을 읽으면 타이밍이 맞는다. 원하면 활용 멘트 — "이 그림도 저절로 다시 돌지 않습니다. 다시 도는 것도 사람이 결정합니다"(↻ 버튼을 누르며).
- **11장 브리지(구두)**: "이 검증 루프를 시작하기 전에, 먼저 Agent가 어떤 데이터와 권한을 사용할 수 있는지 확인해야 합니다."

## 12장 · 데이터 경계

- **한 문장**: 구분 기준은 접속 방법이 아니라 데이터의 처리·저장 위치다.
- **풀어서**: 오해 교정 두 개를 반드시 입으로 — "사내망에서 접속했다고 사내에서 처리되는 게 아니고, 브라우저로 쓴다고 무조건 외부인 것도 아닙니다." 애매하면 사내 경로 우선.
- **주의**: 이 슬라이드는 보안정책 선언이 아니다 — "구체 허용 범위는 회사 보안정책·승인된 서비스 기준" 각주를 언급. (오해 ③)

## 14장 · 사용 전 3문장 점검

- **한 문장**: 권한? 사외 반출? 검증? — 세 번 다 "예"일 때 도구를 켠다.
- **풀어서**: 체크 문구를 소리 내어 — "외부 메일로 보내도 되는가? 아니오라면 사외 AI에도 넣지 않습니다." 청중이 사진 찍어 가는 장.

## 16장 · 1단계 prompt — 두 공백

- **한 문장**: prompt만으로 데모 1까지는 간다 — 하지만 기준이 매번 다르고(절차), 시스템에 못 닿는다(연결).
- **풀어서**: 데모 1을 "1단계"로 소급 명명 — "우리는 이미 1단계를 봤습니다. 이제 한 단계씩 채웁니다."

## 17–18장 · 데모 2 (2단계 Skill)

- **풀어서**: "정의할 수 있으면 맡길 수 있다"가 축. 18장 셸에서 skill 자동 로드 줄과 unknown 3건 줄을 짚는다. 절차 5번("불확실하면 unknown — 추정하지 않는다") = 근거 없는 추정을 절차로 막는 장치(환각 상세는 부록 44장).
- **주의**: 구성된 예시 라벨. "매번 동일한 JSON"이 아니라 "같은 절차·같은 구조"로만.

## 19장 · Skill 개념

- **한 문장**: Skill은 매번 프롬프트를 길게 쓰지 않기 위한 작업 매뉴얼 — 3장 정의(반복 업무의 절차와 기준)와 같은 말.
- **주의**: 로드 방식은 클라이언트마다 다를 수 있다 — "자동으로 다 된다"처럼 말하지 않는다. 항상 적용되는 프로젝트 지침 / 필요할 때 로드되는 Skill / 연결하는 MCP 구분을 한 번.

## 20장 · Skill 구조 (신설 v1.9 — 구 20장 분리 ①)

- **한 문장**: Skill은 SKILL.md와 필요한 자료로 구성된다 — 핵심 절차는 SKILL.md에, 상세 자료·도구는 필요에 따라 분리.
- **풀어서**: "먼저 Skill이 실제로 어떻게 생겼는지 보겠습니다. SKILL.md가 중심이고, 상세 규칙이나 실행 파일은 필요할 때 분리합니다." 왼쪽 트리(폴더명 = Skill name, 필수는 SKILL.md 하나 — references/scripts/assets는 선택)를 먼저, 오른쪽 해부(① name=식별·폴더명과 일치 → ② description=언제 쓰는 Skill인지 → ③ 본문 절차 → ④ 참조 조건과 경로) 순서로 읽는다.
- **주의**: 이 장은 "어떻게 생겼는가"만 — 실행 과정은 다음 장. 모든 Skill이 네 폴더를 갖는 것처럼 말하지 않는다(선택 폴더는 없을 수 있음). 화면 SKILL.md는 데모 2 Skill의 축약 예시(구성된 예시).

## 21장 · Skill 실행 과정 (신설 v1.9 — 구 20장 분리 ②, 원칙 ③ 배지)

- **한 문장**: 이름·설명으로 찾고 → 선택된 본문을 읽고 → 조건에 해당할 때만 reference를 추가로 읽는다 — 근거가 없으면 unknown.
- **도입 멘트**: "18장에서 Skill이 자동으로 로드되는 장면을 봤습니다. 이제 그 한 줄 안에서 실제로 어떤 일이 일어났는지 천천히 풀어보겠습니다."
- **풀어서**: "처음부터 모든 Skill의 본문과 reference를 읽는 것이 아닙니다. 이름과 설명으로 관련 Skill을 찾고, 선택된 Skill의 본문을 읽은 뒤, 조건에 해당할 때 필요한 reference만 추가로 읽습니다." 세 경로 — A `ASSERT_DMA_LEN` = 기본 절차만으로 assertion 분류·evidence 포함(reference 안 읽음) / B `FLEXLM checkout failed` = unknown 후보 → SKILL.md의 조건 문장 → signature-patterns.md 대조 → env + evidence(파일의 규칙이 결과에 반영되는 장면 — 원칙 ③ 근거를 남긴다를 여기서 짚기) / C `NEW_FATAL_SIGNATURE` = "reference를 읽어도 근거가 없으면 추정하지 않고 unknown으로 남깁니다."
- **주의**: references 폴더가 있다고 자동으로 다 읽는 것이 아니다 — SKILL.md에 적힌 조건과 경로에 따라 읽는다. references는 Skill과 함께 관리하는 자료 — 계속 변하는 외부 시스템 조회는 별도 연결이 필요하다(MCP 상세는 여기서 시작하지 않는다). scripts/assets는 이번 실행에서 사용하지 않음.
- **연출**: 진입 시 1회 ~7.4초 — 요청(0.3s) → 후보·선택(1.0/1.5s) → 본문 로드(2.2s) → 사례 A(3.1s) → 사례 B unknown 후보 + 조건 문장 하이라이트(4.1/4.5s) → reference 열림·규칙 적용(5.1/5.7s) → 사례 C(6.3s) → 요약 스트립(6.9s). ↻로 재생 가능 — lead를 읽고 나서 각 캡션을 순서대로 짚으면 타이밍이 맞는다.

## 22장 · Skill 만드는 법 (신설 v1.9 — 구 20장 분리 ③)

- **한 문장**: 직접 쓰거나, 실제 작업에서 추출한다 — 첫 초안보다 중요한 것은 실제 입력으로 실행하고 고치는 과정.
- **풀어서**: "처음부터 완벽한 Skill 문서를 쓰려고 하지 않아도 됩니다. 실제 작업을 Agent와 함께 수행하고, 내가 준 기준과 수정 내용을 재사용 가능한 절차로 추출하는 방식이 더 현실적입니다." / "다만 생성된 파일은 그대로 신뢰하지 않고 diff를 검토하고 실제 입력으로 시험해야 합니다."
- **전체 요청문(화면은 핵심 행만 — 필요하면 이대로 읽기)**: "방금 수행한 regression triage 과정에서 재사용 가능한 절차를 Skill로 추출해줘. 포함할 내용: name과 description / 입력과 출력 / 실행 절차 / 추정하지 말아야 할 조건 / evidence와 검증 방법 / 실패하거나 판단이 어려울 때의 처리. 상세 규칙은 references/로 분리하고, SKILL.md에 각 파일을 읽는 조건과 경로를 명시해줘. 파일을 생성한 뒤 변경된 파일 목록과 diff를 보여줘."
- **주의**: "맥락 없이 Skill 하나 만들어줘"라는 인상을 주지 않는다 — 실제 업무·입력·출력·예외·사람의 수정 내용이 재료다. 하단 체크리스트(description만 읽어도 용도 / 구체적 입출력 / 실제 순서 / 추정 금지·unknown / reference 조건·경로 / 실제 입력 실행 / diff 사람 검토)를 손으로 짚는다.
- **23장 브리지(구두)**: "이제 반복 절차와 프로젝트 규칙은 Skill에 담았습니다. 하지만 과거 Jira 이슈처럼 계속 바뀌는 정보를 지금 조회하려면, Skill 밖의 시스템과 연결해야 합니다."

## 23–24장 · 데모 3 (3단계 MCP)

- **풀어서**: 같은 시나리오의 다음 질문 — "unknown 3건, 전에도 있었나?" 없을 때 = 연결이 사람(복붙 반복) / 있을 때 = jira_search 한 번. 매칭 없는 1건이 "신규 가능성 — 사람 확인"으로 남는 것까지 짚는다.
- **주의**: 자동화된 건 **조회·대조까지** — 원인 확정·티켓 갱신은 사람. 이슈 ID는 가상.

## 25장 · MCP 개념

- **한 문장**: MCP는 Agent와 데이터·도구를 연결하는 공용 규격(3장 정의 그대로) — 손을 달아주는 것이지 판단력을 보장하는 것이 아니다.
- **주의**: "어디서나"가 아니라 "지원하는 클라이언트에서는". MCP는 보안 정책이 아니다 — 인증·권한은 별도 설계. (오해 ④)

## 26장 · MCP 통제 (v1.9: CORE — 위험 3종만 화면에, 전체 표는 읽기 모드)

- **한 문장**: Agent는 권한을 가진 자동화 실행기 — 권한 설계가 곧 안전 설계.
- **풀어서**: prompt injection만 한 번 풀어 준다("문서·이슈 내용이 지시처럼 작동할 수 있다 — tool 결과는 불신 데이터로"). 마지막에 "두 공백이 채워졌다 — 남은 질문: 이 출력, 파이프라인에 넣어도 되나?"로 본문 3 브리지.

## 29–30장 · Harness란 / UVM 나란히

- **한 문장**: 이 자료에서 Harness는 Agent 실행을 감싸는 권한·검증·재시도·로그·승인 구조 — 검증 환경의 testbench·harness와 비슷한 관점(같다고 말하지 않기), 대상이 Agent 실행.
- **주의**: "믿고 쓸 수 있게"가 아니라 "통제하게" — 양쪽 다 최종 판단은 사람.

## 31장 · Harness 구성 요소

- **한 문장**: 스키마 하나가 아니라 통제 장치들의 묶음 — 믿기 위한 장치가 아니라 통제하기 위한 장치. (오해 ⑥)
- **풀어서**: v1.1 추가 항목을 짚는다 — evidence·confidence 필드(근거·불확실성을 출력에), read-only 기본·dry-run(쓰기 전 diff는 사람이).

## 32장 · 스키마 게이트

- **한 문장**: 구조가 맞다 ≠ 판단이 맞다 · 실행이 끝났다 ≠ 업무가 완료됐다.
- **우리 일로**: "assertion이 violation을 잡아도 spec 해석은 별도 리뷰인 것과 같다." (오해 ⑤)

## 33장 · 언제 쓰나

- **풀어서**: 5질문 체크를 그대로 읽는다 — 자동 입력되나 · 영향 크나 · 되돌리기 어렵나 · 반복되나 · 쓰기 권한 쓰나. 하나라도 "예"면 Harness.

## 34장 · 자율성 단계 L0–L5

- **한 문장**: 처음부터 L5로 가면 사고도 자동화된다 — 시작은 L1~L2.
- **주의**: **외부 표준이 아니라 이 자료의 실무 적용 예시**임을 말한다(슬라이드에 명시됨). v1.8.1 재정의 — L3=읽기 전용 조회, L4=변경안 생성+사람 승인 후 반영, L5=제한된 자동 실행(모니터링·중단·복구). 데모 매핑은 1=L1, 2=L2, 3=L3까지.

## 35장 · 데모 4 (스키마 검증 유무)

- **주의**: 이 데모가 보여주는 것은 **스키마 검증 유무**다 — 축은 형식 일관성·검증 가능성이고, 품질·재현성을 증명하는 데모가 아니다. 내용의 정확성은 별도(근거·사람 검토). 성능 수치·퍼센트 금지. bad run 3종은 벤더 문서가 인정하는 실제 현상.

## 36장 · 회고

- **풀어서**: "오늘 우리는 데모 1의 작업 하나를 실무 시스템 직전까지 키웠습니다" — 단계 여정을 한 호흡으로 복기 + 4원칙 콜백.

## 37장 · 좋은/나쁜 요청

- **한 문장**: 작게 · 구체적으로 · 검증 가능하게. 좋은 요청 3개는 청중이 복사해 갈 문장 — 그대로 읽는다.

## 38장 · 설계 템플릿 9칸

- **한 문장**: 코드가 아니라 글 아홉 줄 — 이게 Skill 초안이고, 승인·실패 줄이 Harness 요구사항이다.
- **풀어서**: "각자 업무 하나를 떠올리고, 이 아홉 칸을 채워 보세요"가 실습 유도 멘트. 오른쪽 triage 예시를 한 줄씩 짧게.

## 39장 · 첫 실험 CTA

- **한 문장**: 첫 목표는 자동화가 아니라, 내가 검증 가능한 반복 작업 하나를 줄이는 것.
- **풀어서**: 5단계 실험 절차(하나 고르기→입력 3개→비교→오류 기록→결정)를 "내일 할 일"로 읽는다. 지표 4개(수정 횟수·누락·오분류·근거 없는 주장)면 충분.
