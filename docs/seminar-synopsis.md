# AI Agent 기술 세미나 — 발표 시놉 (v1.3)

## 기본 정보

- **목표**: AI agent를 처음 소개하고, 팀원들이 "나도 써봐야겠다"는 동기를 갖게 하기
- **청자**: 반도체 설계/검증 담당자. AI agent 이해도 낮음
- **발표 시간**: **90분**
- **핵심 서사(척추)**: *"알고, 써보자"* — AI agent가 무엇이고 → 우리 환경에서 어떻게 쓰고 → 사내에서 첫걸음까지
  - **(v0.9 확정)** "일하는 방식을 바꿔야 한다"(변화 설득)는 척추가 아님 — 주장하지 않고 청중이 결론내리게 하며, **클로징 미래형 한 줄로만** 남김. 표지 제목도 A 행동 선언("오늘부터 쓸 수 있습니다").
- **업무환경**:
  - 사내 직접 서빙 서비스: OpenCode · Roo · DS assistant API — 데이터가 회사 경계를 벗어나지 않음
  - 사외 Claude (도입 예정) — 입력이 외부 서버로 전송
- **자료형태**: reveal.js 단일 파일 HTML(vibe 테마) / 키보드 네비게이션 / autoshow·fragment — **40장** (발표 38장 + 부록 2장)

```

본문 2  확장 요소를 이해한다    → 17 간지 · 18 맨몸 모델의 한계(두 구멍) · 19–20 skill(개념/구조·동작) · 21–22 데모2(skill)
                                  · 23–24 MCP(개념/구조·동작) · 25 데모3(MCP 연결) · 26 도구 지도+설치·주의
본문 3  틀로 감싸 제대로 쓴다   → 27 간지 · 28 실패의 현실 · 29 harness란 · 30 UVM 나란히 · 31 스키마 게이트
                                  · 32 장단점 · 33 언제 쓰나 · 34 데모4(harness 유무)

부록    (발표 시 넘기지 않음)   → 39 도구 후보 목록 · 40 용어집
```

본문은 **우리 환경 → 확장 요소 → 틀** 순으로 난이도 우상향. 본문 1~3은 간지로 시작.
(※ **v1.3: plugin은 발표에서 제외** — 확장 요소 = skill·MCP 둘. 본문 2는 요소마다 "개념 → 구조·동작 → 데모"로 완결. MCP는 약어라 대문자.)

### 확장의 설계 원리 (v1.0, 2026-07-07)

- **A 서사 구멍 해소**: 문제→해결 순서(맨몸 한계 먼저) · 신뢰 캘리브레이션 조기화(실패의 현실) · 용어 선행 정의(LLM 기초) · 데모의 중간 고리(SKILL.md 실물) · CTA 구체화(첫 프롬프트)
- **B 압축 해체**: skill·MCP 각 2장(개념/구조·동작) — 입문 페이스 확보 (역사는 v1.2에서 1장 재병합)
- **C 도메인 공명**: 설계·검증의 반복 pain, UVM 병렬 다이어그램, 사내 도구·판단 사례
- "많이 만들고 줄이기" — 이후 리허설에서 압축 시 페이지리스트에서 삭제 이력 관리

## 도입 — "AI agent란 게 있다" (1~12장, 약 26분)

1. **표지·목차** (1–2장) — 목차는 5부 여정 + "낯선 단어는 그 자리에서 풀어서 + 부록 용어집" 안심 문구
2. **역사** (3장) — 7노드 타임라인(2012 AlexNet → 2024–25 agent loop). 두 시대("말하는 AI"/"일하는 AI") 구분은 lead, "불과 2~3년" 복선은 note
3. **LLM 기초** (4장) — 다음 말 예측(틀릴 수 있음) · 컨텍스트(유한, 넣어준 것만) · 토큰(사용량 단위)
4. **데모 1 (Ex1)** (5장) ⭐ — show→tell. *"regression 로그 실패 분류"* 정적 장면(구성된 예시 라벨) — 대화 6버블 + 실패 분류 리포트 5행
5. **데모 1 해부** (6장) — 세 장면을 핵심 3요소로 매핑
6. **정의** (7장) — "방금 보신 것처럼—" 콜백. "도구 사용 · 다단계 진행 · 결과 확인"
7. **챗봇 vs agent** (8장) — 같은 LLM, 차이는 도구+루프
8. **왜 지금 → 왜 우리 → 재배치 → 위임 기준** (9–12장) — 일반 근거 → 도메인 pain → 역할 재배치(기술 톤, 척추 A) → "결과를 내가 바로 확인할 수 있는가" + 본문 1 브리지

## 본문 1 — 우리 환경 = 정보 경계 (13~16장, 약 12분)

- **격리 다이어그램** (14장): 사용자 노드 두 갈래 분기 — 사내 AI agent(초록) / 경계 벽 관통 → 사외 AI agent(경고색). **보안 원칙 1회 온전 진술** + 이후 콜백만
- **사내 도구 셋** (15장): OpenCode(터미널) · Roo(VS Code) · DS assistant API(직접 호출). 접속 안내 placeholder(`______`)
- **판단 사례** (16장): 사내 경로 vs 사외 경로. **"정확한 기준은 팀 보안 규정이 우선" 명시**

## 본문 2 — 확장 요소: skill · MCP (17~26장, 약 27분) — v1.3: plugin 제외, "개념→구조·동작→데모" 배열

1. **맨몸 모델의 한계** (18장) — **두 구멍**: 지식(절차 모름)·연결(시스템에 못 닿음) → "절차는 skill, 연결은 MCP — 하나씩, 데모와 함께"
2. **skill** (19–22장) — 개념(없으면/있으면 + 매뉴얼 비유) → 구조·동작(파일트리 + progressive disclosure) → **데모 2**: 사람의 일 강조 박스 + 작성법 박스 + SKILL.md 실물(21장) → 실행 flow·달라진 것 3가지(22장)
3. **MCP** (23–25장) — 개념(N×N → USB-C, OpenAI·Google 채택) → 구조·동작(client–server + ①②③, note는 데모 브리지) → **데모 3 (Ex5, v1.3 신설)**: 이슈 트래커 연결 장면(tool 호출 로그 + 상태별 정리, 구성된 예시 라벨). "skill=어떻게, MCP=어디까지" 관계 진술 흡수
4. **도구 지도 + 설치·주의** (26장) — 3칸 지도 × 예 1개, 설치 한 줄, 주의 한 줄(일부 MCP 서버·도구는 외부 연결 → 보안 콜백)

## 본문 3 — harness (27~34장, 약 18분)

1. **실패의 현실** (28장) — 환각 · 들쭉날쭉한 형식 → "확인 자체를 틀에"
2. **harness란** (29장) — 순서·규칙+검증+재시도. 공식 근거 "agentic harness"
3. **UVM 나란히** (30장) — sequence→DUT→assertion ↔ 입력·절차(skill)→모델→스키마 검증
4. **스키마 게이트** (31장) — 형식 게이트 + 정직한 한계(내용은 사람이)
5. **장단점 → 언제 쓰나** (32–33장) — 품질·재현성(동일 구조) vs 비용 → "사람에게 가면 맨 agent, 프로그램에게 가면 harness"
6. **데모 4 (Ex4)** (34장) — 정점. harness 유무 3회 대조(실제 깨진 출력 스냅샷), "같은 구조"까지만
