# AI Agent 기술 세미나 — 발표 시놉 (v1.2)

## 기본 정보

- **목표**: AI agent를 처음 소개하고, 팀원들이 "나도 써봐야겠다"는 동기를 갖게 하기
- **청자**: 반도체 설계/검증 담당자. AI agent 이해도 낮음
- **발표 시간**: **90분** (v1.0에서 60→90분 확정 — 대확장의 근거)
- **핵심 서사(척추)**: *"알고, 써보자"* — AI agent가 무엇이고 → 우리 환경에서 어떻게 쓰고 → 사내에서 첫걸음까지
  - **(v0.9 확정)** "일하는 방식을 바꿔야 한다"(변화 설득)는 척추가 아님 — 주장하지 않고 청중이 결론내리게 하며, **클로징 미래형 한 줄로만** 남김("그 하나가 쌓이면, 일하는 방식은 어느새 바뀌어 있을 겁니다"). 표지 제목도 A 행동 선언("오늘부터 쓸 수 있습니다").
- **업무환경**:
  - 사내 직접 서빙 서비스: OpenCode · Roo · DS assistant API — 데이터가 회사 경계를 벗어나지 않음
  - 사외 Claude (도입 예정) — 입력이 외부 서버로 전송
- **자료형태**: reveal.js 단일 파일 HTML(vibe 테마) / 키보드 네비게이션 / autoshow·fragment — **41장** (발표 39장 + 부록 2장)

```

                                  · 7 정의 · 8 챗봇 vs agent · 9 왜 지금 · 10 왜 우리 · 11 일의 재배치 · 12 무엇을 맡기나
본문 1  우리 환경부터           → 13 간지 · 14 격리 다이어그램 · 15 사내 도구 셋 · 16 보내도 되는 것·안 되는 것
본문 2  확장 요소를 이해한다    → 17 간지 · 18 맨몸 모델의 한계 · 19–20 skill(개념/구조·동작) · 21–22 MCP(개념/구조·동작)
                                  · 23 plugin · 24 관계 정리 · 25 데모2a(SKILL.md 실물) · 26 데모2b(Ex2 실행) · 27 도구 지도+설치·주의
본문 3  틀로 감싸 제대로 쓴다   → 28 간지 · 29 실패의 현실 · 30 harness란 · 31 UVM 나란히 · 32 스키마 게이트
                                  · 33 장단점 · 34 언제 쓰나 · 35 데모3(Ex4)
마무리  회고 → 첫걸음 → Q&A    → 36 오늘 지나온 길(회고) · 37 FAQ · 38 첫걸음 CTA(+첫 프롬프트) · 39 Q&A 클로징
부록    (발표 시 넘기지 않음)   → 40 도구 후보 목록 · 41 용어집
```

본문은 **우리 환경 → 확장 요소 → 틀** 순으로, 뒤로 갈수록 난이도가 우상향하는 상승 구조. 본문 1~3은 간지(디바이더)로 시작.
(※ 확장 요소 = skill·MCP·plugin을 아우르는 우산 용어. 정준 나열 순서 **skill · MCP · plugin**(능력→연결→배포), MCP는 약어라 대문자.)

### v1.0 확장의 설계 원리 (2026-07-07, 20장→대확장)

- **A 서사 구멍 해소**: ① 문제→해결 순서 복원(맨몸 한계가 확장 요소보다 먼저) ② 신뢰 캘리브레이션 조기화("agent도 틀린다"가 harness의 존재 이유) ③ 용어 선행 정의(LLM 기초 — 컨텍스트·토큰을 쓰기 전에 정의) ④ 데모의 중간 고리(SKILL.md 실물 — 입력→skill→출력 3단 완성) ⑤ CTA 구체화(첫 프롬프트 실물)
- **B 압축 해체**: skill·MCP 각 2장(개념/구조·동작), 관계 정리 독립 — 입문 페이스 확보 (역사는 v1.2에서 다시 1장으로 병합)
- **C 도메인 공명**: 설계·검증의 반복 pain, UVM 병렬 다이어그램(sequence→DUT→assertion ↔ 입력·절차→모델→스키마 검증), 사내 도구·판단 사례
- "많이 만들고 줄이기" 전략 — 이후 리허설에서 압축 시 페이지리스트에서 삭제 이력 관리

## 도입 — "AI agent란 게 있다" (1~12장, 약 26분)

1. **표지·목차** (1–2장) — 목차는 5부 여정 + "낯선 단어는 그 자리에서 풀어서 + 부록 용어집" 안심 문구
2. **역사** (3장, v1.2에서 다시 1장 병합) — 7노드 타임라인: 2012 AlexNet → 2022 ChatGPT("말하는 AI") + 2023 function calling → 2024 MCP → 2024–25 agent loop("일하는 AI"). 두 시대 구분은 lead 한 줄, "갖춰진 게 불과 2~3년" 복선은 note. 강조는 '무엇이 가능해졌나', 용어는 괄호 보조
3. **LLM 기초** (4장) — 오늘 쓸 개념 셋: 다음 말 예측(계산 아닌 예측 → 틀릴 수 있음) · 컨텍스트(유한한 작업 기억) · 토큰(사용량 단위). 뒤의 확장 요소·token 절약 도구가 전부 이 장에 기댐
4. **데모 1 (Ex1)** (5장) ⭐ — 정의보다 앞(show→tell). *"regression 로그 실패 분류"* 장면(팀 용어에 맞춰 영문 표기) — 청중의 일을 소재로. 좌 대화 6버블 + 우 "실패 분류 리포트" 5행, **정적 표시(v1.2에서 자동 재생 애니메이션 제거)**. 슬라이드에 "구성된 예시 장면" 라벨 명시
5. **데모 1 해부** (6장) — 세 장면(표 정리 제안/리포트 성장/피드백 반영)을 핵심 3요소로 매핑 — 정의 장의 정지작업
6. **정의** (7장) — "방금 보신 것처럼—" 콜백. 2칸 대조 + "도구 사용 · 다단계 진행 · 결과 확인" ('검증' 단어는 도메인 충돌로 회피)
7. **챗봇 vs agent** (8장) — 같은 LLM, 차이는 도구+루프. 말로 끝 vs 작업물로 끝
8. **왜 지금 → 왜 우리 → 재배치 → 위임 기준** (9–12장) — 일반 근거(9) → 도메인 pain(10: 스펙·문서/로그·리포트/코드·스크립트) → 역할 재배치(11, 기술(記述) 톤 — 척추 A 유지) → 무엇을 맡기나(12: "결과를 내가 바로 확인할 수 있는가" + 민감 데이터 → 본문 1 브리지)

## 본문 1 — 우리 환경 = 정보 경계 (13~16장, 약 12분)

- **격리 다이어그램** (14장): 사용자 노드 두 갈래 분기 — 위(초록) 사내 AI agent(OpenCode · Roo · DS assistant API, 직접 서빙) / 아래(경고색) 빗금 "회사 경계" 벽 관통 → 사외 AI agent(Claude, 도입 예정). 사용자 = 무엇을 어디로 보낼지 결정. **보안 원칙은 여기서 1회만 온전히 진술** — 이후 장은 콜백만
- **사내 도구 셋** (15장): OpenCode(터미널) · Roo(VS Code) · DS assistant API(직접 호출) — 당장 실제로 열어볼 대상의 실물 소개. 접속 안내는 placeholder(`______`)
- **판단 사례** (16장): 원칙을 행동으로 — 사내 경로(업무 자료 활용, 권한 경계 유지) vs 사외 경로(RTL·스펙·미공개 금지). **"정확한 기준은 팀 보안 규정이 우선" 명시** — 슬라이드가 보안 정책을 창설하지 않음

## 본문 2 — 확장 요소: skill · MCP · plugin (17~27장, 약 27분)

1. **맨몸 모델의 한계** (18장) — 문제 제기: 지식(절차 모름)·연결(시스템에 못 닿음)·공유(포장 단위 없음) → "절차는 skill, 연결은 MCP, 공유는 plugin"
2. **skill 2장** (19–20장) — 개념(없으면/있으면 대조 + "잘 쓰는 사람의 매뉴얼" 비유) / 구조·동작(파일트리 + progressive disclosure ①②③)
3. **MCP 2장** (21–22장) — 개념(N×N 문제 → USB-C 표준, OpenAI·Google 채택) / 구조·동작(client–server + ①②③ + 사용 장면 예)
4. **plugin** (23장) — 배포 단위. plugin.json+skills/hooks/+.mcp.json, marketplace, 꼬리표
5. **관계 정리** (24장) — 능력·연결·배포 3칸 + "plugin은 그릇" + 18장 세 구멍과 1:1 대응
6. **데모 2 — 2장으로** (25–26장) — 25장 SKILL.md 실물(frontmatter+절차 5줄, "코드가 아니라 우리말 절차서") → 26장 실행(회의 메모 → 스키마 완전 일치 JSON). 도메인 다리: "설계·검증에선 스펙 문서·검증 로그가 이 자리에"
7. **도구 지도 + 설치·주의** (27장) — 3칸 지도 × 예 1개, 설치 한 줄, 주의 한 줄(외부 연결 → 보안 콜백). harness 브리지는 간지(28장)가 담당

## 본문 3 — harness : "틀로 감싸서 제대로 쓴다" (28~35장, 약 18분)

1. **실패의 현실** (29장, 정직한 전제) — agent도 틀린다: 그럴듯한 오답(환각) · 들쭉날쭉한 형식. "매번 사람이 확인하면 자동화 의미가 없다 → 확인 자체를 틀에" — harness의 존재 이유를 서사로 성립시킴
2. **harness란** (30장) — 순서·규칙+검증+재시도의 실행 틀. **공식 근거**: Claude Code 문서의 "agentic harness". '항상 같은 품질' 절대 표현 금지
3. **UVM 나란히** (31장, 도메인 공명의 정점) — 두 패널 병렬: sequence→DUT→assertion·checker vs 입력·절차(skill)→모델→스키마 검증. "구조가 같고 대상만 칩에서 모델로"
4. **스키마 게이트** (32장) — 통과/실패(한도 내 재시도 ↻). 스키마=assertion처럼 **형식** 게이트, 내용 정합은 별도(지금은 사람이 — CTA 사람 검수 루프와 연결)
5. **장단점 → 언제 쓰나** (33–34장) — 품질·재현성(동일 **구조**) vs 설계·유지보수 비용 → 판단 기준 "출력이 사람에게 가면 맨 agent, 프로그램에게 가면 harness"
6. **데모 3 (Ex4)** (35장) — 정점. harness 없이(3회 모두 다른 실패 — 실제 깨진 출력 스냅샷 노출) vs 감싸면(3회 모두 스키마 일치, "같은 구조" — 값 결정론 표현 금지). 실패 3종은 벤더 문서가 인정하는 실제 현상

## 마무리 — 회고 → FAQ → 첫걸음 → Q&A (36~39장, 약 12분)

1. **오늘 지나온 길** (36장, 회고) — 어디서(데이터는 사내에)·무엇으로(확장 요소 셋)·어떻게(틀로 감싸서) 3칸 요약 + fragment 펀치라인 "남은 건 써 보는 것"
2. **FAQ 선제 대응** (37장) — Q1 내 일이 없어지나(없어지는 건 반복) · Q2 어디까지 믿나(형식은 틀, 내용은 사람) · Q3 뭘 공부하나(일을 정의하는 연습)
3. **첫걸음 CTA** (38장, "그래서, 어떻게 시작하나") — 사내 LLM에서 · 스스로 판단 가능한 작은 작업 · 사람 검수 루프 + **첫 프롬프트 실물**(복붙 가능한 한 문단 — "불확실은 ?로 표시" 포함)
4. **Q&A 클로징** (39장) — A 착지 + B 지평선 한 줄. 지지대 칩: 이 자료 · 부록 · 옆자리 동료
