# AI Agent 기술 세미나 — 발표 시놉 (v0.9)

## 기본 정보

- **목표**: AI agent를 처음 소개하고, 팀원들이 "나도 써봐야겠다"는 동기를 갖게 하기
- **청자**: 반도체 설계/검증 담당자. AI agent 이해도 낮음
- **핵심 서사(척추)**: *"알고, 써보자"* — AI agent가 무엇이고 → 우리 환경에서 어떻게 쓰고 → 사내에서 첫걸음까지
  - **(v0.9 확정)** "일하는 방식을 바꿔야 한다"(변화 설득)는 척추가 아님 — 주장하지 않고 청중이 결론내리게 하며, **클로징 미래형 한 줄로만** 남김("그 하나가 쌓이면, 일하는 방식은 어느새 바뀌어 있을 겁니다"). 표지 제목도 B 선언("일하는 방식이 바뀝니다")에서 A 행동 선언("오늘부터 쓸 수 있습니다")으로 교체.
- **업무환경**:
  - 사내 직접 서빙 서비스: OpenCode · Roo · DS assistant API — 데이터가 회사 경계를 벗어나지 않음
  - 사외 Claude (도입 예정) — 입력이 외부 서버로 전송
- **자료형태**: reveal.js 단일 파일 HTML(vibe 테마) / 키보드 네비게이션 / autoshow·fragment — **19장** (발표 18장 + 부록 1장)

## 전체 구조 (한눈에)

```
도입    AI agent란 게 있다      → 1 표지 · 2 역사 · 3 데모1(Ex1) · 4 정의 · 5 왜 지금
본문 1  우리 환경부터           → 6 우리 환경 = 정보 경계 (1장 병합)
본문 2  확장 요소를 이해한다    → 7 skill · 8 MCP · 9 plugin (각 1장) · 10 데모2(Ex2) · 11 도구+설치·주의
본문 3  틀로 감싸 제대로 쓴다   → 12 harness란 · 13 스키마 게이트 · 14 장단점 · 15 데모3(Ex4)
마무리  정점 → 첫걸음 → Q&A    → 16 다시 그 장면 · 17 월요일 CTA · 18 Q&A 클로징
부록    (발표 시 넘기지 않음)   → 19 확장 요소 후보 목록 (자가학습용)
```

본문은 **우리 환경 → 확장 요소 → 틀** 순으로, 뒤로 갈수록 난이도가 우상향하는 상승 구조.
(※ 확장 요소 = skill·MCP·plugin을 아우르는 우산 용어. 정준 나열 순서 **skill · MCP · plugin**(능력→연결→배포), MCP는 약어라 대문자.)

## 도입 — "AI agent란 게 있다" (1~5장)

1. **AI의 역사** (2장)
   - 6개 구체 마일스톤(연도·명칭, 공식 출처 검증): 2012 AlexNet → 2017 Transformer → 2020 GPT-3(few-shot) → 2022 ChatGPT(RLHF) → 2023 function calling → 2024–25 MCP·agent
   - **(v0.9)** 각 칸의 강조를 '용어명'이 아니라 **'무엇이 가능해졌나'**로, 용어는 괄호 보조(예: "문맥을 통째로 보는 구조(self-attention)")
   - 정밀도: instruction tuning·RLHF의 확립은 InstructGPT(2022), ChatGPT는 대중화 — 구두 설명으로. MCP(연결 표준)와 자율 루프(agent 구조)는 별개 층으로 분리 표기.

2. **데모 1 (Ex1)** (3장) ⭐ — **정의보다 앞** (v0.9 순서 변경: show → tell)
   - *"이 발표자료도 AI로 만들었다"* — 이 슬라이드를 만든 대화 과정 애니메이션 (대화 재생 + 구조 성장, 자동 재생)
   - 역할: 훅 + "이런 게 가능하다"는 실물 증거. 바로 다음 장(정의)이 "방금 보신 것처럼—"으로 받음
   - 청중 노출 라벨은 **데모 1** (내부 관리번호 Ex1 유지)

3. **AI agent의 정의** (4장)
   - 데모 콜백으로 시작. 기존 방식 vs agent 방식 **2칸 대조** + 하단 요약 띠 "핵심 세 가지 — 도구 사용 · 다단계 진행 · 결과 확인"
   - ('검증' 단어는 청중의 도메인 용어 verification과 충돌하므로 정의에서는 '결과 확인'으로)

4. **왜 지금 우리가** (5장)
   - **(v0.9)** 제목의 "지금"에 실제로 답함: 도구 호출(2023)·연결 표준(2024)이 갖춰져 최근에야 실무 투입 가능 + 사내 LLM이 이미 있음
   - 카드: "지금까지(반복 작업에 시간)" vs "agent를 쓰면(판단·설계에 시간)" — 수동형 '받아들이면' 폐기
   - 끝 줄이 본문 1 브리지: "이미 쓸 수 있는 환경이 회사 안에 있습니다 — 먼저 우리 환경부터"

## 본문 1 — 우리 환경 = 정보 경계 (6장, v0.9에서 1장으로 병합)

- 구 "두 갈래" + "정보는 어디로" 2장이 같은 2분할을 반복 → **경계 다이어그램 1장**으로 병합
- **(2026-07-07) 카드 2장 → 격리 다이어그램으로 시각화**: 좌 "회사 안(격리된 사내망)" 존 — 사용자 노드에서 화살표 두 갈래 분기 / 중앙 빗금 "회사 경계" 벽 / 우 "회사 밖" 존. 위 갈래(초록): 사내 AI agent 노드(OpenCode · Roo · DS assistant API, 직접 서빙)로 — 데이터가 사내에 머문다. 아래 갈래(경고색): 벽을 관통해 사외 AI agent 노드(Claude)로 — 코드·데이터가 경계를 넘는다
- 구분 축은 제품명이 아니라 **데이터가 어디로 가는가**:
  - 사내에 머문다 — 사내 직접 서빙 서비스(OpenCode · Roo · DS assistant API), 데이터가 경계를 벗어나지 않음
  - 경계를 넘는다 — 사외 Claude(도입 예정), 입력이 회사 밖 서버로 전송 — 나간 데이터는 회사가 통제 불가
- 사용자 노드가 "무엇을 어디로 보낼지 결정하는 사람" — 데이터 관리 책임이 사용자에게 있음을 도식 안에서 표현
- **보안 원칙은 여기서 1회만 온전히 진술**: "민감·설계 데이터는 사내에서. 판단이 서지 않으면 보안 검토 먼저." 이후 장은 콜백만(재진술 금지)

## 본문 2 — 확장 요소: skill · MCP · plugin (7~11장)

1. **개념 — 각 1장씩** (7·8·9장, 2026-07-07 확장)
   - 공통 템플릿: 의미(제목)·목적(lead) + 좌 "구조" / 우 "동작 원리"(①②③ 스텝)
   - **7장 skill**: 재사용할 절차·전문지식을 파일로. 구조=폴더 하나(SKILL.md+참조 자료, 파일트리). 동작=필요할 때만 펼치기(①이름·설명만 상주 ②맞으면 본문 로드 ③참조 파일은 그때만 — 공식 비유 "목차→챕터→부록 매뉴얼")
   - **8장 MCP**: 외부 연결 개방형 표준("AI용 USB-C"). 구조=client–server, 서버가 tools(도구)·resources(데이터)·prompts(워크플로) 노출. 동작=①연결·tool 목록 교환 ②모델이 골라 호출(로컬·원격) ③결과 수신. 근거 보강: OpenAI·Google도 채택한 공용 규격
   - **9장 plugin**: skill·MCP·hook을 묶은 배포 단위(의미·목적·구조만 — 동작원리 대신 설치·이름표). 구조=plugin.json+skills/hooks/+.mcp.json(파일트리). marketplace 설치 시 plugin명:이름 꼬리표. 관계 진술(꾸러미 vs 담기는 것)은 이 장 하단
   - 신규 용어는 우리말 풀이 먼저, 원어는 괄호. stdio/HTTP 등 전송명은 입문에 과해서 미노출

2. **데모 2 (Ex2)** (10장) — 회의 메모 → 스키마 JSON 구조화 (기본 동작)
   - **(v0.9)** JSON을 선언 스키마와 완전 일치시킴(task·decisions 포함, due는 ISO 통일) — 데모 3의 "완결" 기준이 되므로
   - rule에 도메인 다리 반 줄: "설계·검증에선 이 자리에 스펙 문서·검증 로그가 들어갑니다"

3. **유용한 것들 + 설치 + 주의** (11장, v0.9에서 1장으로 병합)
   - 3칸 카테고리 지도(Context 확장 / 웹·브라우저 / Token 절약) × 예 1개씩 — 전체 후보는 **부록(19장)**
   - 설치는 한 줄(설정에 항목 등록 수준), 주의도 한 줄(일부 MCP·plugin은 외부 연결 → 본문 1 원칙 콜백)
   - 끝 줄이 본문 3 브리지: "이제 이 확장 요소들을 틀로 감싸 제대로 쓰는 법으로"

## 본문 3 — harness : "틀로 감싸서 제대로 쓴다" (12~15장)

1. **harness란** (12장)
   - 확장 요소를 정해진 순서·규칙으로 묶고, 출력을 검증하고, 실패하면 재시도하는 실행 틀
   - **청중 언어**: DUT를 감싸는 verification harness(UVM)와 같은 구조, 대상이 모델일 뿐
   - **(v0.9) 공식 근거 명시**: Claude Code 문서가 스스로를 "agentic harness"라고 부름 — 공식 용어임을 한 줄로
   - '항상 같은 품질' 같은 절대 표현 금지 → "검증되고 재현되는 실행"

2. **스키마 게이트** (13장)
   - 입력→skill 실행→스키마 검증→통과(출력)/실패(한도 내 재시도, 실패 원인 되먹임 ↻)
   - **정직한 한계 명시**: 스키마 = assertion처럼 **형식**을 지키는 게이트. 내용의 정합은 별도 검사 — 그 자리를 지금은 사람이 맡음(마무리 CTA의 '사람 검수 루프'와 연결)

3. **장점과 단점** (14장) — 품질(검증 통과분만 출력)·재현성(동일 **구조**) vs 초기 설계·유지보수 비용

4. **데모 3 (Ex4)** (15장) — 클라이맥스
   - 같은 작업을 harness 없이(3회 모두 다른 실패) vs 감싸면(3회 모두 스키마 일치)
   - **(v0.9) 재현성 표현 교정**: "매번 동일한 JSON"(값 결정론 함의) → "매번 스키마에 일치 / 같은 구조". 우측에 메커니즘 미니 라벨 "(스키마 검증 → 불합격 시 재생성)" 노출
   - 실패 3종(필드 누락/키 드리프트/파싱 불가)은 벤더 문서가 인정하는 실제 현상 — 정직한 illustrative

## 마무리 — 정점 → 안전한 첫걸음 → Q&A (16~18장)

1. **다시 그 장면** (16장, 정서 정점)
   - 데모 1을 **정지컷 1장**으로 (재생 X). 대화 3버블은 데모 1과 **글자 그대로 동일**(콜백 동일성)
   - lead는 **한 문장**으로 다이어트: "처음엔 마법처럼 보이던 이 화면, 이제는 메커니즘으로 보이시죠."

2. **그래서 월요일 아침** (17장, 핵심 CTA)
   - 딱 하나 집어줌: 사내 LLM에서(데이터 사내 유지) · 결과를 스스로 바로 판단할 수 있는 작은 작업(스펙 구조화·로그 요약) · **사람이 최종 확인하는 루프**(harness는 도입되면, 지금은 그 자리를 사람이)
   - "애매하면 보안 먼저" — 본문 1 원칙 콜백

3. **Q&A 클로징** (18장)
   - A 착지 + B 지평선: "이제 저 화면의 원리를 아는 여러분입니다 — 이번 주, 사내 LLM에서 작은 작업 하나로 첫 대화를 시작해 보세요. // 그 하나가 쌓이면, 일하는 방식은 어느새 바뀌어 있을 겁니다."
   - 지지대 칩: 이 자료 · 부록(도구 목록) · 옆자리 동료

## 부록 (19장, 발표 시 넘기지 않음)

- 확장 요소 후보 전체 목록(3칸 × 3개) — 자료 배포 후 자가학습 진입로. 클로징 칩의 '부록' 참조가 실제로 가리키는 장
