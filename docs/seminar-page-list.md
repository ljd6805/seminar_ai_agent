## 부록 A — 데모 2 / 데모 4 공유 parsing 예시 (실제 내용)

> 하나의 "문서 parsing" 작업을 데모 2(skill)·데모 4(harness)가 공유.
> 아래는 **구성된 예시(illustrative)** — 실제로 참인 메커니즘(스키마 검증 + 재시도가 일관성·품질을 높임)을
> 보여주기 위한 대표 사례. 특정 시스템의 측정값이 아님.
> 원하면 사용자가 자신의 harness 실제 실행 결과로 교체 가능(선택).

### 공유 입력 문서 (회의 메모)

```
[회의 메모]
10/24 주간 동기화. 참석: 지훈, 민아, 세영.
- 로그인 API 응답 지연 이슈, 원인은 토큰 갱신 로직으로 확인. 지훈이 금요일까지 수정.
- 신규 대시보드 디자인 시안 3안 중 2안으로 결정. 민아가 다음 주 월요일까지 반영.
- 다음 회의 10/31 오후 3시.
```

- 중립적·누구나 이해 가능. 설계/검증 특화(UVM·RTL) 아님 → 범용 소개에 적합. (교체 가능)
- 슬라이드 rule에서 도메인 다리 반 줄로 연결("설계·검증에선 스펙 문서·검증 로그가 이 자리에").

### 사용 skill (24장에서 실물 공개)

```
meeting-notes/SKILL.md — name·description frontmatter + 절차 5줄
(제목·날짜·참석자 추출 / decisions / action_items{owner,task,due} / 날짜 YYYY-MM-DD / next_meeting)
```

### 목표 스키마

```
{ title, date, participants[], decisions[], action_items[{owner, task, due}], next_meeting }
```

### 데모 2 — 기본 동작 (25장)

비정형 문서를 skill이 한 번에 구조화. **잘 되는 케이스만** 보여줌. (스키마 완전 일치 — decisions·task 포함, due는 ISO로 통일. 2024-10-24=목요일 기준 금요일=10-25, 다음 주 월요일=10-28 — 날짜 검증됨)

```json
{
  "title": "주간 동기화",
  "date": "2024-10-24",
  "participants": ["지훈", "민아", "세영"],
  "decisions": ["대시보드 시안 2안으로 결정"],
  "action_items": [
    { "owner": "지훈", "task": "토큰 갱신 수정", "due": "2024-10-25" },
    { "owner": "민아", "task": "대시보드 2안 반영", "due": "2024-10-28" }
  ],
  "next_meeting": "2024-10-31 15:00"
}
```

### 데모 3 — MCP 연결 (28장, Ex5)

이슈 트래커 MCP 서버를 연결해 둔 상태에서 "이번 주 내가 맡은 이슈, 상태별로 정리해줘" → agent가 issue_search tool 호출 → 6건 수신 → 상태별 표(진행 중 2 · 리뷰 대기 3 · 막힘 1). **구성된 예시** — 이슈 ID(UI-142 등)는 가상. 가르치는 점: 연결(조회·로그인·복사)이 사람 손에서 사라짐.

### 데모 4 — 같은 작업, harness 유무 (37장, Ex4)

같은 입력을 **3번씩** 돌린다고 상상. 축은 **품질 + 재현성**. (슬라이드에도 아래 스냅샷을 실제 텍스트로 노출)

**harness 없이 (단발 호출, 실행마다 출력이 달라짐)**

- Run 1 — 필드 누락 + 구조 드리프트: `participants`가 배열이 아니라 한 문자열, `action_items`에 `due` 빠짐
  ```json
  { "title": "주간 동기화", "participants": "지훈, 민아, 세영",
    "action_items": ["로그인 API 수정", "대시보드 반영"] }
  ```
- Run 2 — 키 이름이 바뀜(스키마 불일치): 한글 키로 나옴
  ```json
  { "제목": "주간 동기화", "참석자": ["지훈","민아","세영"], "결정": ["2안 채택"] }
  ```
- Run 3 — JSON 깨짐(기계 파싱 불가): 설명 문장이 앞에 붙고 마지막 쉼표 오류
  ```
  다음은 회의 내용을 정리한 결과입니다:
  { "title": "주간 동기화", "date": "2024-10-24", }
  ```

→ **품질**: 누락·형식 오류 발생. **재현성**: 세 번 모두 형태가 다름 → 뒷단에서 받아쓸 수 없음.

**harness로 감싸면 (스키마 검증 + 불합격 시 재생성)**

- Run 1·2·3 — **매번 스키마에 일치**하는 완결 JSON(데모 2와 같은 구조).
  ※ 보장되는 것은 '스키마 일치(구조)'이지 값까지의 결정론이 아님 — 슬라이드도 "같은 구조"로 표기.

### 이 대조가 정직한 이유

- LLM 단발 호출의 스키마 미준수(키 누락·드리프트)와 JSON 앞 설명문(preamble)은 **벤더 공식 문서가 인정하는 실제 현상** — 지어낸 결함이 아님.
- harness의 "스키마 정의 → 검증 → 실패 시 재시도"는 출력을 스키마에 수렴시키는 실제 표준 기법.
- 그래서 위 예시는 **참인 현상을 대표 사례로 보여주는 것**. 단, 특정 harness의 측정 수치인 척은 하지 않음.
- 속도/퍼센트 같은 성능 숫자는 쓰지 않음 — 품질·재현성으로만 승부.
