## 부록 A — Ex2 / Ex4 공유 parsing 예시 설계 (뼈대)

> 하나의 "문서 parsing" 작업을 Ex2·Ex4가 공유.
> 아래 구체 내용은 **뼈대/예시(illustrative)** — 실제 삽입할 값은 추후 확정.

### 공유 입력 문서

- 짧은 비정형 문서 1개. **후보: 회의록 / 이슈 리포트 / 짧은 스펙 발췌** (교체 가능)
- 이번 세미나는 범용 소개가 목적이므로, 너무 설계/검증 특화(UVM·RTL)보다 **누구나 이해할 중립적 문서**가 안전
- 예: 회의록 한 문단 → 제목·날짜·참석자·결정사항·액션아이템으로 구조화되는 성격

### Ex2 — 기본 동작 (본문 2, 페이지 10)

```
[입력]  비정형 문서 (raw text)
   ↓   skill이 parsing
[출력]  구조화 결과 (예: JSON)
        { title, date, participants, decisions, action_items }
```

- 가르치는 점: **비정형 in → 구조화 out**, skill이 하는 일을 눈으로 확인
- 딱 한 번, 잘 되는 케이스만 보여줌 (여기선 실패를 다루지 않음)

### Ex4 — 같은 작업, harness 유무 (본문 3, 페이지 16)

같은 parsing 작업을 두 방식으로 대조. 축은 **품질 + 재현성**.

| | harness 없이 | harness로 감싸면 |
|---|---|---|
| **품질** | 필드 누락·형식 깨짐·JSON 오류 가능 *(예시)* | 검증 게이트 통과, 스키마 일치·완결 |
| **재현성** | 같은 입력을 3번 → 매번 다른 형태 *(예시)* | 3번 돌려도 동일/수렴 |

- 핵심 메시지: harness = 부품을 감싸 **일관되고 신뢰할 결과**를 만드는 틀
- ⚠️ 위 "깨지는 예시"는 **설명용 뼈대**. 실제 대조 화면은 직접 돌린 결과로 채우는 게 정직함 (지어낸 수치·결과 금지)
- ⚠️ 속도/퍼센트 같은 성능 숫자는 쓰지 않기로 함 — 품질·재현성으로만 승부
