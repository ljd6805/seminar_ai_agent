## 부록 A — 관통 시나리오 데모 예시 (실제 내용)

> **하나의 "regression 실패 분류" 작업**을 데모 1(prompt)·데모 2(Skill)·데모 3(MCP)·데모 4(Harness)가 공유.
> 아래는 **구성된 예시(illustrative)** — 참인 메커니즘(스키마 검증+재시도가 일관성을 높임, MCP가 조회를 자동화함)을 보여주기 위한 대표 사례. 특정 시스템의 측정값이 아니며, test명·이슈 ID는 가상.

### 공유 작업 정의 (18·20장 SKILL.md)

```
regression-failure-triage/SKILL.md
- 절차: test·seed·timestamp 추출 → timeout·assertion·build·env·unknown 분류
  → signature 그룹핑 → 대표 로그(evidence) 인용 → 불확실하면 unknown·추정 금지
- 출력: JSON 배열 — category · count · representative_test · evidence · next_action
```

### 데모 1 (4장) — prompt만

대화+기본 파일 도구로 실패 12건을 1차 분류(timeout·assertion은 실패 증상·로그 패턴 기준)하되 **근거 부족 3건은 분류 보류로 사람 검토에 남는다** — 완벽 데모가 아니라 불확실성을 함께 노출. 훅 질문("실무에 넣으려면 무엇이 더 필요한가")의 근거.

### 데모 2 (17–18장) — Skill

정의 → SKILL.md 저장 → opencode 한 줄 실행 → triage.json(timeout 5 / assertion 4 / unknown 3 — evidence 포함) → 주간 실패 현황 자동 조립(가상 시나리오).

### 데모 3 (22–23장) — MCP (예시 시스템 = Jira)

"unknown 3건 — 과거 유사 실패를 Jira에서 찾아줘": 없을 때 = 사람이 로그인→signature 검색→복사→붙여넣기. 있을 때 = jira_search 호출 → 유사 이슈 2건 대조(riscv_dbg_t7≈INFRA-1201 env 원인 / usb_iso_t2≈VERIF-88 seed 재현 / 1건 매칭 없음→사람 확인). 가르치는 점: 자동화된 건 조회·대조까지 — 원인 확정·티켓 갱신은 사람.

### 데모 4 (34장) — Harness 유무

같은 triage 요청을 3번씩 돌린다고 상상. 축은 **품질 + 재현성**.

**harness 없이** — Run 1 필드 누락(`evidence·next_action` 없음) / Run 2 키 드리프트(`"유형"·"건수"` 한글 키) / Run 3 JSON 깨짐(preamble 문장 + 쉼표 오류).
**harness로 감싸면** — 매번 스키마 일치(데모 2의 Skill 출력과 같은 구조). ※ 보장은 '스키마 일치(구조)'이지 값 결정론이 아님.

### 이 대조가 정직한 이유

- 스키마 미준수(키 누락·드리프트)와 preamble은 **벤더 공식 문서가 인정하는 실제 현상**.
- "스키마 정의 → 검증 → 실패 시 재시도"는 실제 널리 쓰이는 기법.
- 속도/퍼센트 등 성능 숫자는 쓰지 않음 — 품질·재현성으로만.

### 보조 예시 — 회의 메모 (부록 49장으로 이동)

구 데모 2의 meeting-notes SKILL.md(제목·날짜·참석자 / decisions / action_items / YYYY-MM-DD / "?" 표시)는 "같은 규격, 다른 소재" 1장으로 유지 — 본편 중심 사례로는 쓰지 않음.
